# 코딩테스트 공부법

> TL;DR: 유형별 학습 → 풀이 → 사후 분석 루프. 1시간 막히면 해설로 전환, "왜 못 풀었나"를 흡수. 책상 앞에서의 풀이 흐름은 [[풀이순서]].

## 학습 순서

1. **알고리즘 유형 하나 선택** — 한 번에 한 유형씩 깊이 학습 (이론 → 코드 → 예제)
2. **백준 / 프로그래머스 / 리트코드에서 해당 유형 문제 3~5개** 연속으로 풀어 패턴 손에 익히기
3. **사후 분석** (아래 루틴 적용)
4. 1주일 후 비슷한 유형 다시 풀어 30초 안에 떠오르는지 확인 — 안 되면 [[../review-queue]]에 추가

---

## 1시간 룰

```
0~20분: 혼자 풀기
20~40분: 힌트(첫 줄만) 보고 방향 수정
40~60분: 해설 유형/키워드만 확인 후 직접 구현
60분 초과: 완전 해설 보고 패턴 학습으로 전환
  → 같은 유형 2~3문제 추가 풀기
  → weak-points.md에 기록
```

> **목적**: 풀이 능력 향상이 목표. 막히는 시간 낭비 > 패턴 학습 손해.

해설을 볼 때:
1. 어떤 부분에서 막혔는지 먼저 파악
2. 해설에서 막혔던 부분의 해결만 집중적으로 읽기
3. 해설 닫고 직접 다시 구현
4. **"내가 왜 이 생각을 못했지?"**를 파악하고 그 아이디어를 흡수
5. 약 1주일 후 비슷한 유형 다시 풀기 — 목표: **1시간 안에 해결**

---

## 풀이 후 분석 루틴

```
✓ 내가 작성한 코드의 시간/공간 복잡도 확인 + 최적화 여지
✓ 핵심 알고리즘의 복잡도와 어느 상황에서 써야 하는지 일반화
✓ 다른 풀이 방식이 있는가? (공간/시간 트레이드오프)
✓ 다른 사람 코드 리뷰 — 다른 시간복잡도나 접근법 학습
✓ 이 문제의 핵심 패턴 한 줄 요약
✓ 틀린 케이스가 있었다면 → 엣지 케이스 패턴 [[../weak-points]] 기록
✓ 30초 이내로 재현할 수 있나? (없으면 [[../review-queue]] 추가)
```

핵심 질문: **"풀면서 했던 생각들을 일반 규칙으로 만들 수 있는가?"**
예: "그리디 문제는 데이터 정렬로 시작" → 다음 그리디 문제에서 30초 단축.
