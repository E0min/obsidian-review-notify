# 출력 조건 구조 파악

> TL;DR: 구현 전 출력이 a/~a · a/b · 다중 중 어떤 구조인지 판별. 여사건이면 실패 조건 발견 즉시 early return.

---

## 핵심 질문

> "출력 조건 두 가지가 있을 때, **완전한 탐색**이 필요한가, 아니면 **한 쪽만 찾으면** 나머지는 자동으로 결정되는가?"

코딩 전에 **출력이 어떤 구조인지** 먼저 파악하면 구현 방향이 달라진다.

---

## 출력 구조 3가지

| 구조 | 특징 | 전략 |
|------|------|------|
| **a / ~a (여사건)** | Yes/No, true/false — 한 쪽이 성립하면 나머지 자동 결정 | 실패 조건 발견 즉시 **early return** |
| **a / b (독립 두 값)** | 두 결과가 서로 다른 조건에서 계산됨 | 두 분기 모두 명시적 처리 |
| **a / b / c / … (다중 값)** | 복수 케이스, 범주 분류 필요 | switch or 조건 체계화 |

---

## a / ~a 코딩 템플릿

"Yes가 되는 조건" 전부 나열하는 대신, **"No가 되는 조건 하나**"를 발견하면 즉시 종료.  
실패 조건이 성공 조건보다 항상 단순하다.

```javascript
function solution(...) {
    for (...) {
        if (실패_조건) return "No";  // 즉시 탈출
    }
    return "Yes";  // 끝까지 살아남으면 성공
}
```

핵심: 성공 조건 전체를 직접 증명하려 들지 말 것. 실패만 빼면 나머지는 성공.

---

## a / ~a 적용 문제 유형

- **완주/통과 여부** — 카드 뭉치 ([[../problems/programmers/159994-카드-뭉치]]), 완주하지 못한 선수
- **유효성 검증** — 괄호 짝맞추기, 회문 판별
- **경로 존재 여부** — BFS/DFS 도달 가능 여부
- **조건 만족 가능성** — 그래프 색칠, 위상 정렬 가능 여부

---

## a / b 적용 문제 유형

두 결과가 별도 분기로 계산되는 경우:

- "성공이면 X, 실패면 -1 반환"
- "조건 만족 시 카운트 +1, 불만족 시 += 비용"
- 두 영역의 합/차이를 별도로 계산해야 하는 경우

```javascript
function solution(...) {
    if (조건A) return 결과A;
    return 결과B;  // 두 분기 모두 명시
}
```

---

## a / b / c … (다중 출력)

3가지 이상의 결과 — switch / 매핑 테이블로 체계화.

```javascript
function classify(x) {
    if (조건1) return "case1";
    if (조건2) return "case2";
    if (조건3) return "case3";
    return "default";
}
```

가능하면 **객체 매핑**으로 분기 줄이기:
```javascript
const handler = {
    type1: (x) => calc1(x),
    type2: (x) => calc2(x),
};
return handler[x.type]?.(x) ?? defaultResult;
```

---

## Pitfalls

- **a / ~a 구조인데 a 직접 증명 시도** → 시간 낭비. 실패 조건만 찾아 early return.
- **a / b 구조에서 한 분기 누락** → 함수 끝에서 `undefined` 반환 → 디버깅 지옥.
- **다중 출력을 if 사슬로** → 가독성 ↓. 매핑 객체나 switch 활용.
