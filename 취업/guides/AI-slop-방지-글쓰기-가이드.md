# AI slop 방지 글쓰기 가이드

## 목표

자기소개서, 이력서, 포트폴리오 Case 본문에서 생성형 AI 특유의 매끈하지만 빈 문장을 줄인다. 기준은 문장이 멋있어 보이는지가 아니라, 읽는 사람이 실제 경험과 판단 과정을 떠올릴 수 있는지다.

좋은 문장은 다음 네 가지 중 하나를 포함해야 한다.

- 숫자: 기간, 횟수, 개선 전후, 처리량, 응답 시간, 사용자 수
- 행위: 설계했다, 구현했다, 측정했다, 비교했다, 대조했다, 배포했다
- 대상: 어떤 파일, 화면, 기능, 데이터, 사용자 흐름을 다뤘는지
- 결과: 오류 감소, 시간 단축, 유지보수 범위 축소, 사용자 혼선 감소

## 금지 문체 패턴 5종

### 1. 명언식 한 줄 선언 남발 금지

금지:

> 확장성은 신뢰의 또 다른 이름이다.

문제:

- 근거 없이 그럴듯하다.
- 기술 경험보다 문장 솜씨가 앞에 나온다.
- 자기소개서에서는 실제 행동이 아니라 감상처럼 읽힌다.

대체:

> 트래픽이 몰릴 때도 응답 시간이 흔들리지 않도록 캐시 계층을 추가하고, P95 응답 시간을 1.2초에서 280ms로 줄였습니다.

### 2. 연속된 단문 나열 금지

금지:

> 병목이 있었다. 원인은 N+1이었다. 인덱스가 없었다. 쿼리를 고쳤다.

문제:

- 사람이 쓴 회고라기보다 요약 모델 출력처럼 보인다.
- 원인, 과정, 결과의 연결이 끊겨 읽는 사람이 사고 과정을 따라가기 어렵다.

대체:

> 목록 화면에서 응답이 지연되는 원인을 확인한 결과, 댓글 수 조회가 게시글마다 반복되는 N+1 구조였고 관련 컬럼 인덱스도 없었습니다. 집계 쿼리로 조회 경로를 바꾸고 인덱스를 추가해 첫 화면 응답 시간을 줄였습니다.

### 3. "X는 Y의 Z다"식 은유적 정의 금지

금지:

> 캐싱은 확장성의 첫 단추다.

문제:

- 설명처럼 보이지만 실제로는 정보가 없다.
- 기술 선택의 이유, 적용 위치, 결과가 빠진다.

대체:

> 동일한 게시글 목록 요청이 반복되어 서버 호출이 누적됐기 때문에, React Query 캐시 시간을 조정하고 페이지 단위 응답을 재사용하도록 바꿨습니다.

예외:

- 정확한 개념 정의는 허용한다.
- 예: `RLS는 Supabase에서 사용자별 행 접근을 제한하는 보안 정책입니다.`

### 4. "단순히 X가 아니라 Y다" 대조 부정 금지

금지:

> 혁신은 단순히 새로운 기술을 도입하는 것이 아니라, 반복되는 비효율을 해결하는 것이다.

문제:

- 자소서 결론처럼 매끄럽지만 흔하다.
- 무엇을 했는지보다 가치 판단이 먼저 나온다.

대체:

> 수기로 관리하던 근무표와 재고 체크를 앱으로 옮기고, 직원들이 같은 화면에서 스케줄과 재고 변동을 확인하게 했습니다. 이후 스케줄 확인과 마감 재고 점검에 쓰는 시간이 줄었습니다.

### 5. 금지 어휘와 관용구

금지:

- em-dash 남용
- `load bearing`
- `blast radius`
- `escape hatch`
- `sanity check`
- `honest`
- `genuine`
- `actual`
- `real`
- `The thing to internalize:`
- `honest caveat:`
- `The smoking gun:`
- **"박-" 류 한국어 내부 은어**: `박제`·`박혔다`·`박았다`·`박는다`·`박아 두다`·`코드에 박다`·`코드로 박다`·`주석에 박다`·`메모리에 박다`

문제:

- 영어 관용구는 영어권 AI 답변에서 자주 보이는 문체 표식이다.
- 한국어 자기소개서에서는 번역투나 과장된 해설문처럼 보인다.
- "박-" 류는 1인 운영자·하네스 개발자 사이의 내부 은어로, 사외 독자(채용 담당자)에게는 어색한 비유로 읽힌다. 사실 기반 동사(`자동 기록`·`기록했다`·`만들었다`·`사전 마련했다`·`사전 구축했다`·`코드로 연결했다`·`명시했다`·`정리했다`)로 대체한다.

대체 원칙:

- 영어 관용구 대신 실제 조치와 결과를 쓴다.
- 강조 부사 대신 수치, 기간, 비교 대상을 넣는다.
- 도입구 없이 바로 판단 근거를 쓴다.
- "박-" 류 대신 무엇을 어디에 어떤 행위로 두었는지 동사로 명확히 쓴다.

## 추가 금지 패턴

### 세 항목 나열 중 마지막이 앞 둘의 합인 구조

금지:

> 안정성, 확장성, 그리고 신뢰성을 높였습니다.

문제:

- 세 번째 항목이 앞의 두 항목을 포괄해 의미가 흐려진다.
- AI가 문장을 풍성하게 보이게 하려고 붙이는 경우가 많다.

대체:

> 장애 복구 절차를 문서화하고, 반복 조회 쿼리를 캐시해 응답 지연을 줄였습니다.

### 추상어만 있는 결과 문장

금지:

> 이를 통해 업무 효율성과 사용자 경험을 개선했습니다.

대체:

> 직원들이 재고를 다시 확인하는 횟수가 줄었고, 마감 전 확인할 항목이 한 화면으로 정리됐습니다.

### 근거 없는 강도 표현

금지:

> 눈에 띄게 줄였습니다.

대체:

> 반복 조사와 초안 작성에 쓰던 시간을 줄이고, 남은 시간을 원문 대조와 수치 확인에 썼습니다.

수치가 있으면 더 좋다.

> 반복 조사와 초안 작성 시간이 하루 2시간에서 40분 정도로 줄었습니다.

## 허용 원칙

- 숫자, 기간, 기술명으로 채운다.
- 1인칭 행위 동사를 쓴다.
- 형용사와 부사보다 명사구와 동사구를 우선한다.
- 추상 비유보다 실제 한 일과 결과를 쓴다.
- 결론을 먼저 멋있게 쓰지 말고, 행동을 먼저 보여준 뒤 배운 점을 짧게 붙인다.

좋은 동사:

- 설계했다
- 도입했다
- 구현했다
- 측정했다
- 비교했다
- 검증했다
- 분리했다
- 자동화했다
- 문서화했다
- 배포했다

피해야 할 동사:

- 체감했다
- 깨달았다
- 느꼈다
- 기여하고 싶다
- 성장하고 싶다
- 중요하다고 생각한다

위 동사는 완전히 금지하지 않는다. 다만 문항 마지막 한두 문장에만 제한적으로 쓴다.

## Case 본문 점검 체크리스트

- [ ] em-dash 0회
- [ ] 은유형 "X는 Y다" 정의문 없음
- [ ] 단문 3개 이상 연속 구간 없음
- [ ] 수치, 기간, 스택 없는 추상어 문장 없음
- [ ] "단순히 ~가 아니라 ~다" 없음
- [ ] 세 항목 나열 중 마지막이 앞 둘의 합인 구조 없음
- [ ] 모든 Case에 문제, 행동, 결과가 들어 있음
- [ ] 결과가 "개선했다"에서 끝나지 않고 무엇이 달라졌는지 보임
- [ ] 마지막 문장이 교훈문으로 과장되지 않음

```text
- 명언식 한 줄 선언
```

## AI 말투 제거 수정 프롬프트

```text
아래 글을 자기소개서/포트폴리오용으로 다시 다듬어줘.

목표:
- AI가 쓴 것처럼 매끈한 문장을 줄이고, 실제 사람이 경험을 설명하는 톤으로 바꾼다.
- 추상어보다 구체적인 행동, 대상, 결과를 앞에 둔다.
- 문장은 자연스럽게 연결하되, 과장된 명언식 결론은 쓰지 않는다.

반드시 지킬 규칙:
- em-dash 사용 금지
- "단순히 ~가 아니라 ~다" 금지

- 단문 3개 이상 연속 금지

- honest, genuine, actual, real 같은 강조 부사 금지
```

## 포트폴리오 Case 작성 프롬프트

```text
아래 구현 경험을 포트폴리오 Case 본문으로 작성해줘.

형식:
1. 문제 원인
2. 해결 과정
3. 결과

문체 규칙:
- 문제 원인은 실제로 어떤 사용자 흐름, 시스템 경로, 데이터 구조에서 문제가 생겼는지 쓴다.
- 해결 과정은 사용한 기술명과 변경한 구조를 쓴다.
- 결과는 수치가 있으면 수치로 쓰고, 없으면 사용자가 보게 된 변화나 운영자가 하게 된 변화로 쓴다.
- 명언식 문장, 은유적 정의, "단순히 ~가 아니라 ~다"는 쓰지 않는다.
- 단문을 끊어 나열하지 말고 원인과 조치가 연결되게 쓴다.

나쁜 예:
"병목이 있었다. 원인은 N+1이었다. 인덱스가 없었다."

좋은 예:
"게시글 목록에서 댓글 수를 게시글마다 다시 조회하면서 N+1 쿼리가 발생했고, 댓글 테이블의 조회 컬럼에도 인덱스가 없어 첫 화면 응답이 늦어졌습니다."

입력:
- 프로젝트:
- 문제 상황:
- 내가 한 일:
- 사용 기술:
- 결과:
```

## 제시한 금지 항목 평가

사용자가 제시한 5개 금지 문체 패턴은 AI 말투를 줄이는 데 효과가 크다. 특히 명언식 선언, 대조 부정, 영어권 AI 관용구는 자기소개서와 포트폴리오에서 바로 티가 난다.

다만 다음 예외는 둔다.

- "X는 Y다"는 전면 금지보다 은유형 정의만 금지한다. 기술 개념을 정확히 설명하는 정의문은 필요하다.
- 단문은 전면 금지하지 않는다. 강조가 필요한 곳의 한두 문장은 괜찮지만, 3문장 이상 이어지면 요약문처럼 보인다.
- 수치가 없을 때 억지로 숫자를 만들면 더 위험하다. 이 경우에는 관찰 가능한 변화, 비교 대상, 작업 전후 흐름을 쓴다.
- 금지 어휘는 한국어 자기소개서에서는 강하게 금지해도 된다. 특히 영어 도입구와 관용구는 번역투로 보인다.

결론:

이 체크리스트는 AI 말투 제거에 충분히 좋다. 다만 "모든 정의문 금지"와 "모든 단문 금지"처럼 과도하게 적용하면 글이 부자연스러워질 수 있으므로, 은유형 문장과 연속 단문을 잡는 용도로 쓰는 것이 맞다.
