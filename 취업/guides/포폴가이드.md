# 포트폴리오 작성 가이드라인 (Project Portfolio Guideline)

- 이 가이드는 이력서 내 프로젝트 섹션 중, **'문제 해결 역량'**을 강조하기 위한 상세 포트폴리오 페이지 작성법을 설명합니다.
- 채용 담당자가 가장 관심 있게 보는 **"어떤 문제가 있었고, 그것을 어떻게 논리적으로 해결했으며, 그 결과는 어떠했는가?"**에 집중하는 구조입니다.
- **가독성 최우선**: 모든 줄글은 가독성을 위해 **불렛 리스트(Bullet Points)** 형태로 작성하며, 마침표 단위로 끊어줍니다.

## 1. 아키텍처, 데이터 플로우, 시퀀스 다이어그램
**프로젝트의 최상단에 하나만 존재해야 합니다.** 프로젝트의 전체 그림을 시각적으로 보여주는 섹션입니다. 텍스트보다는 이미지를 활용해서 시각화를 하여 보여줍니다.

포함해야 할 내용:
- **전체적인 아키텍처 다이어그램 (Architecture Diagram)**: 프론트엔드, 백엔드, 데이터베이스, 외부 API 등 시스템 구성 요소 간의 관계를 다이어그램으로 작성합니다.
- **전체 데이터 플로우/시퀀스**: 시스템의 큰 흐름을 보여주는 다이어그램이 있다면 이곳에 포함합니다.

## 2. 문제 해결 사례 (Problem Solving Cases)
각 문제는 이력서의 한 줄과 매핑되어야 합니다.

작성 구조:
- **현상**: 어떤 문제가 발생했는가? (예: 응답 속도 저하, 메모리 누수, 데이터 불일치 등)
- **원인 분석**: 왜 그 문제가 발생했는가? **근본적인 원인(Root Cause)**을 분석해야 합니다.
- **해결 과정**: 
    - 가설 설정 -> 검증 -> 최종 해결책의 논리적 흐름.
    - **중요**: 해당 문제 해결 과정에서 특정 데이터 흐름이나 로직 설명이 필요하다면, **반드시 이 부분에 별도의 다이어그램(Flow/Sequence)을 생성하여 첨부**합니다.
    - 모든 문장은 **개조식 불렛 리스트**로 작성합니다.
- **결과**:
    - **정량적 성과**: 수치(%)로 증명 (예: LCP 2.5s -> 0.8s).
    - **정성적 성과**: 사용자 경험 개선 등.

## 3. 배운 점 (Insights)
프로젝트 전체를 통해 배우거나 성장한 점을 기술합니다.

📝 템플릿 복사본 (Copy & Paste)

*(여기에 전체 시스템 다이어그램 이미지를 넣으세요)*
- **Architecture**: 기술 스택 및 구조 요약

## Case 1. [이력서의 문제 해결 한 줄]
### 1. 문제 원인
- (문장은 담백하게, 마침표마다 줄바꿈하여 불렛으로 작성)
- 발생했던 현상에 대해 기술합니다.
- 근본적인 원인을 분석합니다.

### 2. 해결 과정
*(필요 시 여기에 해당 기능의 데이터 플로우/시퀀스 다이어그램 추가)*
- 해결을 위해 시도한 가설이나 접근 방식을 기술합니다.
- 구체적인 구현 내용을 서술합니다.
- 기술적 의사결정의 이유를 밝힙니다.

- **성과**: 수치적 개선이나 명확한 결과물.
- **배운 점**: 해당 문제를 통해 얻은 기술적 통찰.
