# 웹 서비스가 운영되는 과정

> TL;DR: 브라우저 클릭 → DNS → TLS → 로드 밸런서 → 웹 서버 → WAS → DB/캐시 → 응답 → 렌더링. 식당 비유로 7단계. 어디서 병목이 나는지 알면 최적화 지점이 보인다.

---

## 식당 비유 — 전체 흐름

| 단계 | 주체 | 역할 비유 |
|---|---|---|
| 요청 | 프론트엔드 | 손님이 메뉴판 보고 주문 |
| 전달 | 네트워크 / DNS | 주문서가 주방으로 (주소 확인) |
| 접수 | 웹 서버 | 식당 입구 지배인 (예약 확인·안내) |
| 조리 | **WAS** | 주방장 (실제 요리·레시피 수행) |
| 재료 | DB / 캐시 | 냉장고·창고 (식재료 꺼내기) |
| 서빙 | 백엔드 → 프론트 | 완성 요리를 식탁에 |
| 식사 | 프론트엔드 | 손님이 요리 먹음 (화면 출력) |

---

## 1. 프론트엔드 — 사용자 요청 시작

URL 입력 또는 버튼 클릭으로 시작.

- **UI 이벤트 발생**: 사용자가 '로그인' 버튼 클릭
- **클라이언트 측 유효성 검사**: JS가 아이디·비밀번호 형식 1차 확인 (서버 부하 방지)
- **API 호출 (Request)**: `fetch` / `axios`로 HTTP 요청 생성
  - Method: `POST`
  - Payload: `{ "id": "user123", "pw": "********" }`

---

## 2. 네트워크 & 인프라 — 주소 찾기와 보안

- **DNS 조회**: `naver.com` → `123.123.123.123` (IP) 변환
- **TCP/TLS 핸드셰이크**: HTTPS 암호화 통로 수립
- **로드 밸런서**: 여러 서버 중 가장 한가한 서버에 요청 분배

---

## 3. 웹 서버 (Nginx / Apache) — 문지기

- **정적 파일 반환**: 이미지·HTML이면 여기서 응답 끝
- **리버스 프록시**: 동적 처리는 뒤의 **WAS**로 전달 (보안·성능)

---

## 4. WAS (Web Application Server) — 비즈니스 로직

Tomcat·Node.js·Django 등이 본격 처리:

- **인증·인가**: "이 사용자 맞나?", "권한 있나?"
- **비즈니스 로직 실행**: 회원 테이블에서 ID 찾아 PW 비교 등
- **ORM / Query 실행**: SQL 생성해 DB 호출

---

## 5. 데이터베이스 & 캐시

- **DB 조회** (MySQL, PostgreSQL): 디스크 데이터 검색
- **캐시 확인** (Redis): 자주 찾는 데이터는 메모리에서 즉시 응답 (DB까지 안 감)
- **결과 반환**: WAS에게 데이터 전달

---

## 6. 응답 생성 (Response)

- **데이터 패키징**: WAS가 JSON 등으로 패키징
- **HTTP 상태 코드**:
  - `200 OK` (성공)
  - `401 Unauthorized` (인증 실패)
  - `500 Internal Server Error` (서버 오류)
- **전달**: 웹 서버 → 브라우저

---

## 7. 프론트엔드 — 렌더링

- **데이터 수신**: 브라우저가 JSON 수신
- **상태 업데이트**: React/Vue가 변경된 데이터로 화면 재렌더 (Virtual DOM)
- **사용자 확인**: "로그인 성공" 메시지 또는 페이지 이동

---

## 어디서 병목이 생기는가?

| 단계 | 흔한 병목 | 최적화 도구 |
|---|---|---|
| DNS | DNS 응답 지연 | DNS prefetch, CDN |
| TLS | 핸드셰이크 비용 | HTTP/2, TLS 세션 재사용 |
| 웹 서버 | 정적 파일 전송 비용 | CDN, gzip/brotli 압축 |
| WAS | 비즈니스 로직 N+1 쿼리 | 쿼리 최적화, 비동기 처리 |
| DB | 인덱스 누락, 풀 스캔 | 인덱스 추가, 쿼리 플랜 분석 |
| 응답 | 큰 페이로드 | 페이지네이션, GraphQL, 압축 |
| 렌더링 | 큰 번들·메인 스레드 블로킹 | 코드 스플리팅, lazy load |
