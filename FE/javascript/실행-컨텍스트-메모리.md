# 실행 컨텍스트와 메모리 아키텍처

> TL;DR: JS 엔진은 메모리를 **콜 스택**(정적·LIFO)과 **힙**(동적·GC)으로 이원화한다. 함수 호출 시 생성되는 **실행 컨텍스트**가 스택에 쌓이며, 여기엔 렉시컬 환경(변수)·외부 스코프 참조·this 바인딩이 담긴다.

---

## What

자바스크립트 엔진이 코드를 실행할 때 메모리를 어떻게 나누고, 변수와 함수 정보를 어디에 저장하는지 다루는 내부 메커니즘.

---

## How

### 1. 메모리 영역 분리 — 왜 두 곳으로 나누는가

| | **콜 스택 (Call Stack)** | **힙 (Heap)** |
|--|-------------------------|--------------|
| 특징 | LIFO, 크기 예측 가능 | 동적 할당, 크기 예측 불가 |
| 저장 대상 | 실행 컨텍스트(함수 정보), 원시값, 포인터 | 객체·배열·동적 문자열 |
| 할당 방식 | 스택 포인터(SP) 이동 → O(1) | 빈 공간 탐색 → 상대적으로 느림 |
| 해제 | 함수 종료 시 자동 (SP 역이동) | GC(가비지 컬렉터)가 회수 |

### 2. 실행 컨텍스트(Execution Context) — 스택에 쌓이는 단위

함수가 호출될 때마다 콜 스택에 **실행 컨텍스트(EC)**가 생성된다.

```
전역 EC 생성 → 스택 바닥
  └─ foo() 호출 → foo EC 생성 → 스택 위
       └─ bar() 호출 → bar EC 생성 → 스택 최상단
            bar 종료 → bar EC 제거
       foo 종료 → foo EC 제거
전역 EC (프로그램 종료 시까지 유지)
```

EC의 내부 구조:

```
실행 컨텍스트 (EC)
├── 렉시컬 환경 (Lexical Environment)
│   ├── 환경 레코드 (Environment Record) — 변수·함수 선언 저장소
│   └── 외부 환경 참조 (Outer Reference) → 상위 스코프 포인터 (스코프 체인)
└── this 바인딩 — 현재 this가 가리키는 객체 주소
```

### 3. 메모리 확보 원리 — 파싱과 스택 포인터

```
함수 호출
  ↓
엔진 Pre-scan: 함수 내부 선언문 훑기 → 슬롯 수 확정
  (변수 5개 → 5 × 8바이트 = 40바이트 필요)
  ↓
SP(스택 포인터)를 40바이트 위로 이동 → 공간 확보 완료
  ↓
변수 초기화 및 코드 실행
  ↓
함수 종료 → SP를 아래로 내림 → 공간 반납 (LIFO)
```

### 4. 재할당과 스택

```javascript
var c = 2;
c = 3; // c 슬롯의 비트 데이터를 덮어씀 (새로 쌓지 않음!)
```

값이 바뀌는 것은 **이미 확보된 슬롯 안을 수정**하는 것 — LIFO 구조를 건드리지 않는다.

### 5. 데이터 타입별 저장 위치 (V8 최적화)

| 타입 | 저장 위치 | 방식 |
|------|---------|------|
| 작은 정수 (Smi) | **콜 스택** | 값 자체를 슬롯에 직접 기록 (포인터 없음) |
| Boolean | **콜 스택** | 값 직접 기록 |
| Object·Array | **힙** | 힙에 실제 데이터, 스택 슬롯에 주소(포인터) |
| String·BigInt | **힙** | 크기 가변 → 힙 저장, 스택엔 주소만 |

## Why it matters

```javascript
function outer() {
  let x = 10;        // outer EC 렉시컬 환경에 저장
  function inner() {
    console.log(x);  // outer EC의 외부 환경 참조를 타고 x를 찾음 (스코프 체인)
  }
  inner();
}
outer();
```

**호이스팅**도 이 원리: Pre-scan 단계에서 `var` 선언과 함수 선언이 먼저 환경 레코드에 등록된다. `let/const`는 TDZ(Temporal Dead Zone)으로 초기화 전 접근 차단.

---

## Pitfalls

- **스택 오버플로**: 재귀 함수가 종료 조건 없이 실행되면 스택 공간이 고갈됨 → `Maximum call stack size exceeded`
- **클로저와 메모리**: 내부 함수가 외부 함수의 EC를 참조하면, 외부 함수 종료 후에도 EC의 환경 레코드가 힙으로 이동해 GC 대상이 되지 않음 → 불필요한 클로저는 메모리 누수 원인
- **V8 Fast mode vs Dictionary mode**: 객체에 프로퍼티를 동적으로 추가/삭제하면 Hidden Class 최적화가 깨져 Dictionary mode로 폴백 → `delete` 남용 주의
