# 웹 성능 측정 (Core Web Vitals + INP)

> TL;DR: 성능 = 비즈니스 (이탈률↓·전환↑) + SEO (검색 순위↑) + UX. Google Core Web Vitals(LCP·CLS·FID·INP)로 측정하고 **측정→개선→재측정** 루프로 개선. 2024년부터 FID 대신 **INP** 중심.

---

## 왜 측정하는가

| 관점 | 효과 |
|---|---|
| **비즈니스** | 느린 페이지 → 이탈률 ↑ → 전환·매출 ↓. 빠른 페이지 → 신뢰감 ↑ |
| **SEO** | Google이 Core Web Vitals를 검색 순위에 반영 |
| **UX** | 빠르고 부드러운 페이지 = 즐거운 사용자 경험 |

---

## 핵심 지표 — Core Web Vitals + INP

### LCP (Largest Contentful Paint)

"페이지가 제대로 로드되기 시작했구나"라고 사용자가 인식하는 순간.
- 보통 **히어로 이미지·큰 타이틀**이 나타나는 시간
- **목표**: 2.5초 이내 (Good), 4초 이내 (Needs improvement)

### CLS (Cumulative Layout Shift)

콘텐츠가 갑자기 튀면서 이동하는 현상의 누적 점수.
- 이미지 로딩 중 레이아웃 밀림·광고 삽입으로 버튼 움직임
- **목표**: 0.1 이하 (Good), 0.25 이하 (Needs improvement)

### FID (First Input Delay) — Deprecated

사용자의 **첫 번째 상호작용**에 브라우저가 반응하기까지 걸리는 시간.
- 2024년 3월부터 INP로 대체

### INP (Interaction to Next Paint) — 신규 표준

페이지 **전체 수명 동안 발생하는 모든 상호작용** 중 가장 느린 반응 시간.
- 첫 클릭만 보던 FID와 달리 전체 인터랙션 품질을 평가
- **목표**: 200ms 이하 (Good), 500ms 이하 (Needs improvement)

## 측정 도구

| 도구 | 용도 | 데이터 종류 |
|---|---|---|
| **WebPageTest** | 서버 응답·리소스 로딩 상세 분석. 네트워크/브라우저 조건 세팅 | Lab Data |
| **PageSpeed Insights** | Google 무료. 점수·개선 가이드. 모바일·데스크톱 분리 | Field + Lab |
| **web-vitals JS 라이브러리** | 실제 사용자 환경에서 LCP·CLS·INP 수집 → 분석 서버 전송 | Field (RUM) |
| **Chrome DevTools — Lighthouse** | 로컬 빠른 측정 | Lab |
| **Chrome DevTools — Performance** | 프레임·태스크 단위 깊은 분석 | Lab |

```javascript
// web-vitals 사용 예
import { onLCP, onCLS, onINP } from 'web-vitals';

onLCP(metric => sendToAnalytics(metric));
onCLS(metric => sendToAnalytics(metric));
onINP(metric => sendToAnalytics(metric));
```

## 개선 방법

### LCP 개선 (가장 큰 요소를 빠르게)

- 메인 배너는 `<img>`로 직접 + `<link rel="preload" as="image">`로 우선 로드
- **SSR / SSG**로 초기 HTML 즉시 렌더
- 이미지 최적화: 필요한 크기만, WebP·AVIF 포맷
- **CDN** 사용 → TTFB 단축

### CLS 개선 (예기치 않은 이동 막기)

- `<img>` 태그에 `width`·`height` 또는 `aspect-ratio` 명시 → 공간 미리 확보
- 광고·동적 콘텐츠 컨테이너 높이 미리 잡기
- 웹 폰트: `font-display: swap` + `size-adjust`로 폰트 교체 시 텍스트 안 튀게

### INP / FID 개선 (메인 스레드 점유 줄이기)

- **코드 스플리팅**: 페이지·기능별 JS 분할 → 초기 로드 ↓
- **롱 태스크(50ms+) 분할**: `requestIdleCallback`, `requestAnimationFrame`, `scheduler.yield()`
- **Web Worker**: 무거운 계산은 별도 스레드로
- **React 최적화**: `useMemo`, `useCallback`, `React.memo`로 불필요한 재렌더 방지
- **React 18+**: `startTransition`, `useDeferredValue`로 우선순위 분리

## Next.js / React 실전 팁

### 1. SSR & SSG

```typescript
// Next.js 14 App Router
// 정적 (SSG)
export default async function Page() {
  const data = await getStaticData();
  return <Article data={data} />;
}

// 동적 (SSR)
export const dynamic = 'force-dynamic';
```

CDN 캐싱으로 TTFB 대폭 개선. LCP 직결.

### 2. 코드 스플리팅 / Dynamic Import

```typescript
// Next.js
import dynamic from 'next/dynamic';

const HeavyChart = dynamic(() => import('@/components/HeavyChart'), {
  loading: () => <Skeleton />,
  ssr: false,  // 클라이언트에서만 로드
});

// React
const Modal = React.lazy(() => import('./Modal'));
```

대시보드·모달 등 초기에 불필요한 컴포넌트를 지연 로드.

### 3. Next.js `<Image>` 컴포넌트

```tsx
import Image from 'next/image';

<Image
  src="/hero.png"
  width={1200}
  height={600}
  priority           // LCP 후보면 priority=true
  placeholder="blur"
/>
```

- 자동 사이즈·WebP 변환·lazy loading
- `priority`로 LCP 이미지 우선 로드
- placeholder로 CLS 방지

### 4. 메모화 & 최적화 훅

```typescript
const expensive = useMemo(() => heavyCalc(items), [items]);
const handler = useCallback(() => doSomething(id), [id]);

const MemoChild = React.memo(Child);
```

복잡 리스트·무거운 계산 컴포넌트에 적용.

### 5. 번들 압축 & 캐싱

- Gzip / Brotli 압축으로 전송량 ↓
- `Cache-Control`·`ETag`로 재방문 시 캐시 활용

## 측정 → 개선 루프

```
1. Lab 측정 (Lighthouse, WebPageTest) — 기준선
2. 가설: "X 컴포넌트가 LCP 늦춤"
3. 개선: priority preload, dynamic import 등
4. 재측정 → 개선 폭 확인
5. Field 측정 (web-vitals + RUM) — 실사용자 데이터 검증
```

성능 최적화는 **한 번에 끝나는 작업이 아니라 반복 프로세스**.

---

## Pitfalls

- **Lab 점수만 보고 안심** — 실제 사용자 환경(저사양 폰·느린 망)은 다름. Field 데이터 필수.
- **INP는 첫 클릭만 보면 안 됨** — 페이지 전체 수명의 모든 인터랙션 평가. 스크롤·드롭다운·SPA 라우팅도 측정 대상.
- **`React.memo` 남발** — 비교 비용이 렌더 비용보다 클 수도. 프로파일러로 실제 효과 확인.
- **이미지 lazy + LCP 후보** — LCP 이미지에 `loading="lazy"` 붙이면 LCP 늦어짐. `priority`/`fetchpriority="high"`로 강제 우선.
- **번들 크기만 보면 안 됨** — 실행 시간(execution time)이 INP에 더 큰 영향. Coverage 탭으로 미사용 코드 확인.
