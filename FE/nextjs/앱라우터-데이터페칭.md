# App Router 데이터 페칭

> TL;DR: App Router에서는 서버 컴포넌트 어느 곳에서나 `async/await`로 직접 fetch해서 props drilling 없이 필요한 곳에 데이터를 바로 주입할 수 있다.

## What

Next.js에서 서버 데이터를 컴포넌트에 전달하는 방식. Page Router와 App Router의 접근법이 근본적으로 다르다.

## Why it matters

Page Router는 `getServerSideProps` / `getStaticProps`를 **페이지 최상단에서만** 실행할 수 있었다. 깊은 곳의 컴포넌트가 데이터를 필요로 하면 최상단 → 중간 → 하위 컴포넌트 순으로 props를 내려줘야 했다(props drilling). 컴포넌트가 많아질수록 유지보수가 어려워졌다.

App Router는 서버 컴포넌트라면 **어디서든** `async/await`로 직접 fetch할 수 있게 해서 이 문제를 해소했다.

## How

### Page Router 방식 (getServerSideProps)

```javascript
// pages/blog/index.js
// 데이터 페칭은 반드시 파일 최상단의 특수 함수에서만 가능
export async function getServerSideProps() {
  const res = await fetch('https://api.example.com/posts');
  const posts = await res.json();
  return { props: { posts } }; // props로만 전달 가능
}

// 최상단 컴포넌트에서 받아서 하위로 내려줘야 함
export default function BlogPage({ posts }) {
  return <PostList posts={posts} />;
}
// PostList → PostItem 순으로 계속 props drilling
```

### App Router 방식 (서버 컴포넌트 직접 fetch)

```javascript
// app/blog/page.js — 서버 컴포넌트에서 직접 fetch
async function Example() {
  const data = await fetch('https://api.example.com/data').then(res => res.json());
  return <div>{JSON.stringify(data)}</div>;
}
```

필요한 컴포넌트 어디서든 동일하게 사용 가능:

```javascript
// app/blog/page.js
import PostList from '@/components/PostList';

export default async function BlogPage() {
  const posts = await fetch('https://api.example.com/posts').then(r => r.json());
  return <PostList posts={posts} />;
}

// components/PostList.js — 이 컴포넌트도 서버 컴포넌트라면 직접 fetch 가능
export default async function PostList() {
  // props drilling 없이 직접 fetch
  const posts = await fetch('https://api.example.com/posts').then(r => r.json());
  return (
    <ul>
      {posts.map(post => <li key={post.id}>{post.title}</li>)}
    </ul>
  );
}
```

### 두 방식 비교

| 항목 | Page Router | App Router |
|------|-------------|-----------|
| 데이터 페칭 위치 | 파일 최상단 특수 함수 | 서버 컴포넌트 어디서나 |
| 전달 방식 | `props` / Context API | 서버 컴포넌트에서 직접 fetch |
| 서버 컴포넌트 지원 | 없음 | 기본 지원 |
| Hydration 비용 | 페이지 전체 필요 | 서버 컴포넌트는 불필요 |
| Props drilling | 발생 | 해소 |
| 데이터 페칭 함수 | `getServerSideProps`, `getStaticProps` | `async function` |

### 병렬 데이터 페칭 (성능 최적화)

```javascript
export default async function Page() {
  // 순차 실행 — 느림
  // const user = await fetchUser();
  // const posts = await fetchPosts();

  // 병렬 실행 — 빠름
  const [user, posts] = await Promise.all([
    fetchUser(),
    fetchPosts(),
  ]);

  return (
    <div>
      <h1>{user.name}</h1>
      <PostList posts={posts} />
    </div>
  );
}
```

### 서버 액션으로 데이터 변경 (쓰기)

```javascript
// app/posts/new/page.js
export default function NewPostPage() {
  async function createPost(formData) {
    'use server'; // 서버 액션 선언
    const title = formData.get('title');
    await db.insert({ title });
    redirect('/posts');
  }

  return (
    <form action={createPost}>
      <input name="title" />
      <button type="submit">작성</button>
    </form>
  );
}
```

## Pitfalls

- 같은 URL을 여러 서버 컴포넌트에서 중복 fetch해도 Next.js가 **자동으로 요청을 중복 제거(deduplication)** 하므로 실제 네트워크 요청은 1회만 발생
- 서버 컴포넌트에서 fetch한 데이터는 기본적으로 캐싱됨 → 실시간 데이터가 필요하면 `cache: 'no-store'` 옵션 필요 (→ [[데이터-캐시]] 참조)
- `async` 서버 컴포넌트와 클라이언트 컴포넌트(`'use client'`)는 동시에 쓸 수 없음 — 데이터 페칭이 필요하면 서버 컴포넌트에서 하고 결과를 props로 전달
