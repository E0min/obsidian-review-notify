# 스트리밍
> TL;DR: 서버가 HTML을 한 번에 보내는 대신 준비된 부분부터 점진적으로 클라이언트에 전송해 사용자가 더 빨리 콘텐츠를 볼 수 있게 하는 기법.

## What
동적 페이지는 풀 라우트 캐시가 없기 때문에 요청마다 서버에서 렌더링한다. 이때 모든 데이터가 준비될 때까지 기다렸다가 한 번에 응답하면 사용자는 오래 기다려야 한다. 스트리밍은 이미 준비된 UI 조각부터 순서대로 클라이언트로 보내 **빈 화면 대기 시간을 줄인다**.

두 가지 방식으로 적용한다.
1. **페이지 스트리밍** — `loading.tsx`로 페이지 단위 로딩 UI 제공
2. **컴포넌트 스트리밍** — `<Suspense>`로 특정 컴포넌트 단위 로딩 UI 제공

## Why it matters
- TTFB(Time To First Byte)가 아니라 사용자가 **실제로 무언가를 볼 수 있는 시간**이 줄어든다.
- 무거운 데이터 요청이 있는 컴포넌트만 늦게 로딩되고, 나머지 UI는 즉시 표시된다.
- 스켈레톤 UI 등 로딩 상태를 세밀하게 제어할 수 있다.

## How

### 페이지 스트리밍 — `loading.tsx`

async 페이지 컴포넌트가 데이터를 가져오는 동안 보여줄 대체 UI를 정의한다.

```typescript
// app/blog/loading.tsx
export default function Loading() {
  return <p>Loading...</p>;
}
```

- `loading.tsx`는 같은 디렉토리 및 하위 디렉토리 페이지 모두에 공통 적용된다.
- **페이지 컴포넌트에만 적용** — 일반(비-async) 컴포넌트에는 효과 없음.
- **쿼리 스트링 변경 시 적용 안 됨** — `/blog?page=2`처럼 URL 파라미터만 바뀔 때는 `loading.tsx`가 트리거되지 않는다.

### 컴포넌트 스트리밍 — `<Suspense>`

비동기 컴포넌트를 `<Suspense>`로 감싸면 해당 컴포넌트만 독립적으로 스트리밍된다.

```typescript
// app/blog/page.tsx
import { Suspense } from 'react';
import AsyncComponent from './AsyncComponent';

export default function BlogPage() {
  return (
    <div>
      <h1>블로그</h1>
      {/* AsyncComponent가 준비되기 전까지 fallback 표시 */}
      <Suspense fallback={<p>Loading...</p>}>
        <AsyncComponent />
      </Suspense>
    </div>
  );
}
```

### 쿼리 스트링 변경 시 Suspense 재트리거

쿼리 파라미터가 바뀔 때도 로딩 UI를 보여주려면 `key` prop을 활용한다.

```typescript
// searchParams가 바뀔 때마다 Suspense를 새로 마운트
<Suspense key={searchParams.query} fallback={<p>Loading...</p>}>
  <SearchResults query={searchParams.query} />
</Suspense>
```

## Pitfalls
- `loading.tsx`는 쿼리 스트링 변경에 반응하지 않는다. 검색·필터·페이지네이션처럼 쿼리 파라미터로 데이터가 바뀌는 경우에는 `<Suspense key={...}>`를 써야 한다.
- `<Suspense>`로 감싸지 않은 비동기 컴포넌트가 하나라도 있으면, 그 컴포넌트가 완료될 때까지 해당 영역 전체가 블로킹된다.
- `loading.tsx`가 하위 디렉토리에 공통 적용되는 특성 때문에, 특정 하위 페이지에서만 다른 로딩 UI가 필요하다면 해당 디렉토리에 별도 `loading.tsx`를 만들어 오버라이드해야 한다.
