# 라우트 세그먼트 옵션
> TL;DR: 페이지·레이아웃 파일에서 export 상수로 선언해 해당 라우트의 렌더링 방식, 캐싱, 런타임을 제어하는 옵션 집합.

## What
App Router의 `page.tsx` 또는 `layout.tsx` 파일 최상단에 특정 상수를 export하면 Next.js가 해당 라우트의 동작을 빌드·요청 시점에 바꾼다. 코드 레벨에서 SSR/SSG/ISR을 명시적으로 제어할 수 있다.

## Why it matters
기본 동작(`auto`)만으로 제어가 안 되는 경우 — 예컨대 특정 페이지를 항상 SSR로 강제하거나, Edge Runtime에서 실행하거나, fetch 캐싱 전략을 통일하고 싶을 때 — 이 옵션들로 세밀하게 조정할 수 있다.

## How

### `dynamic` — 렌더링 방식 강제

```javascript
// SSR 강제: 매 요청마다 새로 렌더링
export const dynamic = 'force-dynamic';

// 정적 강제: 동적 함수가 있어도 무시하고 정적 렌더링
export const dynamic = 'force-static';

// 기본값: Next.js가 자동 판단
export const dynamic = 'auto';
```

### `revalidate` — ISR 재생성 주기

```javascript
// SSR처럼 캐시 없이 매 요청 렌더링
export const revalidate = 0;

// 60초마다 백그라운드에서 페이지 재생성
export const revalidate = 60;

// ISR 비활성 (무한히 캐시 유지)
export const revalidate = false;
```

### `fetchCache` — fetch 캐싱 기본값 통일

```javascript
// 이 라우트의 모든 fetch를 강제 캐싱
export const fetchCache = 'force-cache';

// 이 라우트의 모든 fetch를 캐시하지 않음
export const fetchCache = 'no-store';
```

### `runtime` — 실행 환경

```javascript
// 기본값: Node.js 런타임 (모든 Node.js API 사용 가능)
export const runtime = 'nodejs';

// Edge Runtime: 전 세계 엣지 노드에서 실행, 빠른 응답
// 단, Node.js 전용 API(fs, crypto 등) 사용 불가
export const runtime = 'edge';
```

### 조합 패턴

```typescript
// 정적 우선 ISR: 5분마다 재생성, 항상 캐시에서 반환
export const dynamic = 'force-static';
export const revalidate = 300;
export const fetchCache = 'force-cache';

// SSR 실시간: 매 요청마다 렌더링, Edge에서 빠르게 응답
export const dynamic = 'force-dynamic';
export const revalidate = 0;
export const fetchCache = 'no-store';
export const runtime = 'edge';
```

## Pitfalls
- `force-static`으로 설정하면 `cookies()`, `headers()` 같은 동적 함수의 반환값이 빈 값으로 처리된다. 실제 쿠키나 헤더가 필요한 페이지에 쓰면 버그가 생긴다.
- `runtime = 'edge'`는 Node.js 내장 모듈(fs, path, crypto 등)을 사용할 수 없다. ORM 라이브러리 중 Edge 미지원 패키지가 많으므로 사전 확인이 필요하다.
- `revalidate`는 해당 라우트의 **모든 fetch**에 일괄 적용된다. 개별 fetch마다 다른 주기가 필요하면 fetch 옵션(`next: { revalidate: N }`)으로 따로 지정해야 한다.
