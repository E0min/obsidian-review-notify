# 클라이언트 라우터 캐시
> TL;DR: App Router가 브라우저 메모리에 RSC 페이로드를 저장해 페이지 이동 시 공통 레이아웃 데이터를 재요청하지 않는 클라이언트 측 캐싱.

## What
Next.js App Router는 서버 컴포넌트가 렌더링한 결과(RSC 페이로드)를 브라우저 인메모리에 저장한다. 이후 다른 페이지로 이동할 때 이미 캐시된 레이아웃·공통 컴포넌트 데이터는 재사용하고, 새 페이지의 고유 데이터만 서버에 요청한다.

- 저장 위치: 브라우저 메모리 (탭 내부)
- 수명: 세션 유지 중 (새로고침 또는 탭 종료 시 초기화)
- 적용 대상: `<Link>` 컴포넌트 또는 `router.push()`로 이동 시

## Why it matters
레이아웃처럼 여러 페이지가 공유하는 컴포넌트의 데이터를 매번 서버에서 받아오면 불필요한 네트워크 비용이 발생한다. 클라이언트 라우터 캐시는 이 중복 요청을 없애 **페이지 전환 속도**를 크게 향상시킨다.

## How

**동작 시나리오**

```
1. 사용자가 /dashboard 접속
   → 서버에서 layout.js 데이터 + dashboard/page.js 데이터 수신
   → 두 데이터 모두 클라이언트 라우터 캐시에 저장

2. 사용자가 /settings 로 이동
   → layout.js 데이터: 캐시에서 즉시 재사용 (서버 요청 없음)
   → settings/page.js 데이터: 서버에 새로 요청
```

**캐시 계층 순서**

```
Request Memoization   (렌더링 사이클 내 중복 제거)
        ↓
데이터 캐시           (서버 전역, 거의 영구)
        ↓
풀 라우트 캐시        (서버, 빌드 타임 정적 페이지)
        ↓
클라이언트 라우터 캐시 (브라우저 인메모리, 세션 단위)
```

**캐시 강제 갱신**

```typescript
import { useRouter } from 'next/navigation';

const router = useRouter();
// 클라이언트 라우터 캐시를 무효화하고 서버 컴포넌트 재실행
router.refresh();
```

## Pitfalls
- 새로고침(`window.location.reload()`)하면 클라이언트 라우터 캐시가 전부 초기화된다. 새로고침 후 첫 페이지 이동은 모든 데이터를 다시 서버에서 받는다.
- `router.refresh()`는 서버 컴포넌트를 재실행하지만 클라이언트 컴포넌트의 로컬 상태(useState)는 유지된다. 서버 데이터만 갱신하고 싶을 때 적합하다.
- Next.js 14 이후로 클라이언트 라우터 캐시의 기본 수명이 단축되어, 레이아웃은 5분, 페이지는 30초로 자동 만료된다.
