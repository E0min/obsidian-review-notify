# 풀 라우트 캐시
> TL;DR: 빌드 타임에 정적 페이지의 HTML과 RSC 페이로드를 서버에 미리 생성해 저장하는 캐싱 레이어.

## What
Next.js가 `next build` 시점에 **Static Page**를 미리 렌더링하여 HTML과 RSC 페이로드를 서버 파일시스템에 저장한다. 이후 요청이 들어오면 새로 렌더링하지 않고 캐시된 결과를 즉시 반환한다.

- 적용 대상: Static Page (동적 조건이 없는 페이지)
- 저장 위치: 서버 (데이터 캐시와 달리 빌드 산출물로 존재)
- 수명: 재배포 또는 revalidate 전까지 유지

## Why it matters
매 요청마다 서버 렌더링을 수행하는 대신 미리 생성된 결과를 반환하므로 응답 속도가 매우 빠르다. CDN과 결합하면 전 세계 어디서나 빠른 응답이 가능하다.

## How

**Dynamic Page 조건** — 아래 중 하나라도 해당하면 풀 라우트 캐시가 적용되지 않는다.
1. `cache: 'no-store'` fetch를 사용하는 서버 컴포넌트
2. `cookies()`, `headers()`, `searchParams` 같은 동적 함수를 사용하는 서버 컴포넌트

**동적 경로에 풀 라우트 캐시 적용** — `generateStaticParams`로 빌드 타임에 경로를 생성한다.

```typescript
// app/book/[id]/page.tsx

export async function generateStaticParams() {
  const books = await fetch(`${process.env.URL}/book`).then(r => r.json());
  // 빌드 타임에 모든 도서 ID로 정적 페이지 생성
  return books.map((book: { id: string }) => ({ id: book.id }));
}
```

**정의된 params 외 요청 차단**

```typescript
// generateStaticParams에 없는 경로 → 404 반환
export const dynamicParams = false;
```

**ISR 설정** — revalidate 값으로 일정 시간마다 페이지를 재생성한다.

```typescript
export const revalidate = 60; // 60초마다 재생성
```

## Pitfalls
- `generateStaticParams`를 선언하면 해당 페이지는 강제로 Static으로 설정된다. 동적 함수를 함께 쓰면 충돌이 생긴다.
- `dynamicParams = false` 설정 시 빌드 타임에 정의되지 않은 ID로 접근하면 404가 반환되므로, 데이터가 자주 추가되는 경우 주의가 필요하다.
- 풀 라우트 캐시 재검증 후 첫 요청에서 즉시 캐시가 SET되지 않는다. 실제 사용자가 방문하는 시점에 렌더링이 일어나고 그때 캐시에 저장된다.
