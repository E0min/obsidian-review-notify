# 코드 품질 도구 생태계

> TL;DR: 5개 카테고리로 묶이는 도구들 — 정적분석(ESLint·Prettier), 타입안정성(TypeScript·Zod), 테스트(Vitest·RTL·Playwright), 자동화(Husky·lint-staged·Commitlint), 시각관리(Storybook·Bundle Analyzer).

---

## What

프로덕션 수준 FE 프로젝트에서 "코드 품질"은 단일 도구가 아닌 **도구 체인**으로 보장된다. 각 도구가 담당 영역이 다르며, 자동화 레이어(Husky)가 이들을 커밋/푸시 시점에 강제 실행시킨다.

---

## 5개 카테고리

### 1. 정적 분석 — ESLint · Prettier · Stylelint

```bash
npm install -D eslint prettier eslint-config-prettier @typescript-eslint/eslint-plugin
```

```javascript
// eslint.config.js (ESLint v9 flat config)
import js from '@eslint/js';
import tsPlugin from '@typescript-eslint/eslint-plugin';
import prettier from 'eslint-config-prettier'; // Prettier와 충돌 규칙 비활성화

export default [
  js.configs.recommended,
  {
    plugins: { '@typescript-eslint': tsPlugin },
    rules: {
      '@typescript-eslint/no-unused-vars': 'error',
      '@typescript-eslint/no-explicit-any': 'warn',
    },
  },
  prettier, // 반드시 마지막에 (Prettier 규칙이 ESLint 규칙을 override)
];
```

```json
// .prettierrc
{
  "semi": true,
  "singleQuote": true,
  "tabWidth": 2,
  "trailingComma": "es5",
  "printWidth": 100
}
```

**역할 분리**:
- ESLint: **코드 품질** (미사용 변수, 잠재적 버그, 베스트 프랙티스)
- Prettier: **코드 포맷** (들여쓰기, 따옴표, 줄바꿈) — 팀 내 스타일 논쟁 제거

### 2. 타입 안정성 — TypeScript · Zod

```bash
npm install -D typescript @types/node @types/react
npm install zod
```

```json
// tsconfig.json (Next.js 권장 설정)
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true, // 배열 접근 시 undefined 가능성 강제 인식
    "exactOptionalPropertyTypes": true
  }
}
```

TypeScript는 컴파일 타임 타입 체크, Zod는 **런타임** 데이터 검증. 둘은 보완적.

```typescript
// Zod — API 응답, 폼 데이터, 환경 변수 검증
import { z } from 'zod';

const UserSchema = z.object({
  id: z.number(),
  name: z.string().min(1),
  email: z.string().email(),
});

type User = z.infer<typeof UserSchema>; // TypeScript 타입 자동 추출
```

### 3. 테스트 — Vitest · React Testing Library · Playwright

```bash
npm install -D vitest @vitejs/plugin-react @testing-library/react @testing-library/user-event
npm install -D playwright @playwright/test
```

| 도구 | 담당 | 특징 |
|------|------|------|
| Vitest | 유닛·통합 테스트 | Jest 호환, Vite 생태계 |
| React Testing Library | 컴포넌트 동작 테스트 | DOM 렌더링, 사용자 관점 |
| Playwright | E2E 테스트 | 실제 브라우저, 크로스 브라우저 |

```typescript
// Playwright — E2E 로그인 테스트
import { test, expect } from '@playwright/test';

test('로그인 후 대시보드로 이동', async ({ page }) => {
  await page.goto('/login');
  await page.fill('[name="email"]', 'test@example.com');
  await page.fill('[name="password"]', 'password123');
  await page.click('button[type="submit"]');
  await expect(page).toHaveURL('/dashboard');
});
```

### 4. 자동화 — Husky · lint-staged · Commitlint

```bash
npm install -D husky lint-staged @commitlint/cli @commitlint/config-conventional
npx husky init
```

```bash
# .husky/pre-commit — 커밋 전 staged 파일에만 린트·포맷 실행
npx lint-staged
```

```json
// package.json
{
  "lint-staged": {
    "*.{ts,tsx}": ["eslint --fix", "prettier --write"],
    "*.{css,scss}": ["stylelint --fix", "prettier --write"],
    "*.{json,md}": ["prettier --write"]
  }
}
```

```bash
# .husky/commit-msg — 커밋 메시지 컨벤션 강제
npx --no -- commitlint --edit $1
```

```javascript
// commitlint.config.js
export default {
  extends: ['@commitlint/config-conventional'],
  // 허용 형식: feat: ..., fix: ..., chore: ..., docs: ...
};
```

**자동화 체인 흐름**:
```
git commit →
  pre-commit hook: lint-staged (ESLint + Prettier, staged 파일만) →
  commit-msg hook: commitlint (메시지 형식 검사) →
  커밋 완료
```

### 5. 시각 관리 — Storybook · Bundle Analyzer

```bash
# Storybook
npx storybook@latest init

# Bundle Analyzer (webpack 기반)
npm install -D webpack-bundle-analyzer
# Next.js
npm install -D @next/bundle-analyzer
```

```typescript
// stories/Button.stories.tsx
import type { Meta, StoryObj } from '@storybook/react';
import { Button } from './Button';

const meta: Meta<typeof Button> = {
  component: Button,
  argTypes: {
    variant: { control: 'select', options: ['primary', 'secondary'] },
  },
};
export default meta;

type Story = StoryObj<typeof Button>;

export const Primary: Story = {
  args: { variant: 'primary', children: '저장' },
};
```

```javascript
// next.config.js — Bundle Analyzer
const withBundleAnalyzer = require('@next/bundle-analyzer')({
  enabled: process.env.ANALYZE === 'true',
});
module.exports = withBundleAnalyzer({});
// 실행: ANALYZE=true npm run build
```

**Storybook 용도**: 컴포넌트 독립 개발 + 디자이너 핸드오프 + 시각적 회귀 테스트
**Bundle Analyzer 용도**: 번들 크기 병목 파악 → Tree Shaking·Code Splitting 근거

## 전체 도구 체인 한눈에

```
개발 중:     TypeScript (타입) + ESLint (품질 실시간)
저장 시:     Prettier (포맷 자동)
커밋 시:     lint-staged (staged 파일 lint+format) → commitlint (메시지)
PR 시:       Vitest (유닛), Playwright (E2E) — CI/CD에서 실행
빌드 시:     Bundle Analyzer (사이즈 리포트)
컴포넌트:    Storybook (독립 개발·문서화)
```

---

## Pitfalls

- **ESLint와 Prettier 충돌**: `eslint-config-prettier`를 ESLint 설정 **마지막**에 배치 → Prettier가 담당하는 포맷 규칙을 ESLint가 건드리지 않게
- **lint-staged 대신 전체 린트**: `eslint .`은 전체 파일 → 느림. `lint-staged`는 staged 파일만 → 빠름. pre-commit에는 반드시 lint-staged
- **Storybook 버전 맞춤**: Next.js 15에서 Storybook 8+ 사용 권장. 버전 불일치 시 RSC 지원 문제 발생
- **Playwright 병렬 실행**: CI에서 `--workers=4` 등 병렬화 없으면 E2E가 너무 느림
