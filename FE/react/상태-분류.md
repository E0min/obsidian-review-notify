# React 상태 분류 — 어디에 무엇을 담을 것인가

> TL;DR: 상태를 잘못된 곳에 두면 복잡성이 기하급수적으로 증가한다. Local → URL → Server → Global 우선순위로 배치하고, 나머지는 파생(derived) 상태로 처리한다.

---

## What — 상태의 5가지 종류

```
State
├── Local State         — 컴포넌트 내부 (useState, useReducer)
├── Shared/Global State — 여러 컴포넌트 공유 (Context, Zustand, Jotai)
├── Server State        — 서버 데이터 캐시 (TanStack Query, SWR)
├── URL State           — URL 파라미터/쿼리 (Router, useSearchParams)
└── Form State          — 입력값 + 유효성 (React Hook Form, Formik)
```

---

## 각 분류 상세

### 1. Local State

**정의**: 단일 컴포넌트 또는 그 하위 트리 안에서만 사용되는 상태.

```typescript
// useState — 단순 값
const [isOpen, setIsOpen] = useState(false);
const [count, setCount] = useState(0);

// useReducer — 복잡한 상태 전환
const [state, dispatch] = useReducer(reducer, initialState);
```

**언제**: 다른 컴포넌트가 이 상태를 알 필요가 없을 때. **기본 선택지 — 다른 분류가 필요하다고 확신할 때만 벗어날 것**.

### 2. Shared/Global State

**정의**: 컴포넌트 트리 어디서든 읽고 쓸 수 있는 상태.

```
Context API    — 낮은 빈도 변경 (테마, 언어, 인증 사용자)
Zustand        — 중규모 앱, 빠른 세팅, 선택적 구독
Jotai          — atom 단위 구독, 렌더링 최소화
Redux Toolkit  — 대규모 팀, 엄격한 상태 패턴, DevTools 필수
```

**언제**: prop drilling이 3단계 이상이거나, 서로 다른 subtree에서 같은 상태가 필요할 때.

### 3. Server State

**정의**: 서버에서 가져와 캐시하는 데이터. 항상 outdated될 수 있음.

```typescript
// TanStack Query
const { data, isLoading, error } = useQuery({
  queryKey: ['user', userId],
  queryFn: () => fetchUser(userId),
  staleTime: 5 * 60 * 1000, // 5분 fresh
});
```

**핵심 특성**:
- 비동기 (loading/error/success 상태)
- **서버가 진실의 원천** — 클라이언트는 캐시
- 자동 refetch, background refetch
- optimistic update

**언제**: API 호출이 필요한 모든 데이터. `useState + useEffect + fetch` 조합은 지양.

### 4. URL State

**정의**: URL 자체가 상태. 뒤로가기, 공유, 새로고침 후에도 복원 가능.

```typescript
// Next.js App Router
const searchParams = useSearchParams();
const query = searchParams.get('q');      // /search?q=react
const page = searchParams.get('page');    // /search?q=react&page=2

// 업데이트
const router = useRouter();
router.push(`/search?q=${newQuery}&page=1`);
```

**언제**: 필터, 정렬, 검색어, 현재 탭, 페이지네이션 등 **공유 가능해야 하는 UI 상태**.

### 5. Form State

**정의**: 사용자 입력값 + 유효성 검사 상태. Submit 전까지 임시.

```typescript
// React Hook Form
const { register, handleSubmit, formState: { errors } } = useForm<FormData>({
  defaultValues: { email: '', password: '' },
});
```

**언제**: 입력 필드가 5개 이상이거나 복잡한 유효성 검사가 필요할 때 전용 라이브러리 사용.

## 파생 상태 (Derived State)

**정의**: 기존 상태에서 계산 가능한 값 — 상태로 저장하지 말 것.

```typescript
// ❌ 파생 상태를 별도 state로 저장 — 동기화 버그 위험
const [items, setItems] = useState<Item[]>([]);
const [totalPrice, setTotalPrice] = useState(0); // items 바뀔 때마다 수동 sync

// ✅ 렌더 중 계산
const [items, setItems] = useState<Item[]>([]);
const totalPrice = items.reduce((sum, item) => sum + item.price, 0); // 자동 최신
```

비용이 큰 계산은 `useMemo`로 메모이제이션:

```typescript
const sortedItems = useMemo(
  () => [...items].sort((a, b) => a.price - b.price),
  [items]
);
```

## 상태 배치 결정 플로우차트

```
새로운 데이터를 다뤄야 한다
        ↓
서버에서 오는 데이터인가?
  ├── YES → TanStack Query (Server State)
  └── NO
        ↓
URL에 반영돼야 하는가? (공유, 새로고침 복원)
  ├── YES → URL State (searchParams, router)
  └── NO
        ↓
폼 입력값인가?
  ├── YES → Form State (React Hook Form)
  └── NO
        ↓
다른 컴포넌트도 필요한가?
  ├── NO → Local State (useState/useReducer)
  └── YES
        ↓
얼마나 자주 변경되는가?
  ├── 거의 안 변함 (테마, 언어) → Context API
  ├── 중간 빈도 → Zustand / Jotai
  └── 자주 변함 + 복잡한 로직 → Redux Toolkit
```

## 흔한 실수 — 상태 과잉

```typescript
// ❌ 모든 것을 useState로
const [user, setUser] = useState(null);       // → TanStack Query
const [searchQuery, setSearchQuery] = useState(''); // → URL state
const [isModalOpen, setIsModalOpen] = useState(false); // OK
const [filteredItems, setFilteredItems] = useState([]); // → 파생 상태

// ✅ 제자리 배치
const { data: user } = useQuery({ queryKey: ['user'], queryFn: fetchUser });
const searchQuery = useSearchParams().get('q') ?? '';
const [isModalOpen, setIsModalOpen] = useState(false);
const filteredItems = items.filter(item => item.name.includes(searchQuery));
```

## Pitfalls

- **Server state를 useState에**: 비동기 처리, 캐싱, refetch 모두 수동 — TanStack Query를 쓰면 50줄 짜리 custom hook이 5줄로 줄어든다
- **파생 상태 동기화 버그**: `items`가 바뀌면 `totalPrice`도 수동 업데이트 → 깜빡하면 stale — 계산식으로 대체
- **너무 이른 전역화**: "혹시 다른 데서 쓸 수 있으니" local state를 global로 — 복잡성 증가, lazy하게 올릴 것
- **URL state 미활용**: 필터/정렬을 useState로 → 새로고침 시 초기화, 공유 불가
