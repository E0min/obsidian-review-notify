# 폰트 및 이미지 적용

> TL;DR: 폰트는 `public/`에 두고 CSS `@font-face`로 참조. 이미지는 `public/`(경로 문자열)과 `src/assets/`(import, 번들러 최적화) 두 가지 방식이 있으며, 성능 면에서 `src/assets/`가 유리하다.

## What

React 프로젝트(Vite/CRA)에서 커스텀 폰트와 이미지를 적용하는 두 가지 방법 — `public/` 폴더 직접 참조 vs `src/assets/` import.

## Why it matters

방법에 따라 번들러 최적화(캐싱, 해시 파일명) 적용 여부가 달라진다. 잘못 선택하면 배포 후 캐싱 문제나 경로 오류가 발생할 수 있다.

## How

**폰트** — `public/` 폴더에 파일 저장 후 CSS:

```css
/* index.css */
@font-face {
    font-family: "NanumPenScript";
    src: url("/NanumPenScript-Regular.ttf");
}
body * { font-family: "NanumPenScript"; }
```

**이미지 — public vs assets**:

```javascript
// public/ → 경로 문자열로 직접 참조 (번들러 캐싱 없음)
<img src={"/some-asset.png"} />

// src/assets/ → import 필요, 브라우저 캐싱 최적화
import someAsset from "./assets/some-asset.png";
<img src={someAsset} />
```

**여러 이미지 import 최적화** — 유틸 함수 패턴:

```javascript
// util/getEmotionImage.js
import emotion1 from "../assets/emotion1.png";
import emotion2 from "../assets/emotion2.png";

export default function getEmotionImage(id) {
    switch(id) {
        case 1: return emotion1;
        case 2: return emotion2;
        default: return null;
    }
}

// 사용
<img src={getEmotionImage(1)} />
```

## Pitfalls

- `public/`은 빌드 최적화 안 됨 → 파일명 해시 없음, 캐시 무효화 어려움
- `src/assets/`는 번들러가 파일명에 해시 추가 → 캐시 관리 자동화
- 폰트 파일(`.ttf`, `.woff2`)은 `public/`에 두고 `/` 절대 경로로 참조하는 것이 일반적
- `src/assets/` import 경로 오타 시 빌드 에러 발생 (런타임이 아닌 빌드 타임 검출)
