# 상태관리 라이브러리 비교

> TL;DR: Zustand(중간 규모 빠른 세팅), Jotai(atom 단위 렌더링 최적화), Redux Toolkit(대규모 팀·엄격한 규칙), Context API(외부 라이브러리 없이 테마·언어만). 규모와 팀 상황으로 선택.

---

## What

컴포넌트 간에 공유해야 하는 상태를 어디에, 어떻게 관리할지 결정하는 라이브러리·패턴 선택 문제. prop drilling 없이 전역/공유 상태를 읽고 쓸 수 있게 해준다.

---

## How

### A. Zustand — 가장 빠르게 세팅, 직관적 API

```bash
npm install zustand
```

```typescript
import { create } from 'zustand';
import { persist, devtools } from 'zustand/middleware';

interface CounterState {
  count: number;
  increment: () => void;
  decrement: () => void;
  reset: () => void;
}

// 스토어 정의
const useCounterStore = create<CounterState>()(
  devtools(
    persist(
      (set) => ({
        count: 0,
        increment: () => set((state) => ({ count: state.count + 1 })),
        decrement: () => set((state) => ({ count: state.count - 1 })),
        reset: () => set({ count: 0 }),
      }),
      { name: 'counter-storage' } // localStorage 키
    )
  )
);

// 컴포넌트에서 선택적 구독 (리렌더링 최소화)
function Counter() {
  const count = useCounterStore((state) => state.count);       // count만 구독
  const increment = useCounterStore((state) => state.increment); // 액션만 구독
  return <button onClick={increment}>{count}</button>;
}
```

**특징**:
- 보일러플레이트 최소 (Redux 대비 70% 적음)
- `persist` 미들웨어로 localStorage 동기화 한 줄
- 선택적 구독으로 불필요한 리렌더링 방지
- devtools 미들웨어로 Redux DevTools 연동

### B. Jotai — atom 단위 렌더링 최적화, 복잡한 UI 상태

```bash
npm install jotai
```

```typescript
import { atom, useAtom, useAtomValue, useSetAtom } from 'jotai';

// 기본 atom
const countAtom = atom(0);
const nameAtom = atom('홍길동');

// 파생 atom (derived) — 다른 atom을 읽어 계산
const doubledCountAtom = atom((get) => get(countAtom) * 2);

// 비동기 atom
const userAtom = atom(async () => {
  const res = await fetch('/api/user');
  return res.json();
});

// 컴포넌트 — atom 단위로 구독 → 해당 atom 변경 시만 리렌더
function Counter() {
  const [count, setCount] = useAtom(countAtom);
  const doubled = useAtomValue(doubledCountAtom); // 읽기 전용

  return (
    <div>
      <p>count: {count} / doubled: {doubled}</p>
      <button onClick={() => setCount((c) => c + 1)}>+1</button>
    </div>
  );
}
```

**특징**:
- atom 단위 구독 → 관련 없는 atom 변경 시 리렌더 없음
- 파생 atom으로 Recoil 스타일 데이터 흐름
- React Suspense 친화적 (비동기 atom 내장)
- 스토어 없음 → 테스트 격리 쉬움

### C. Redux Toolkit (RTK) — 대규모 팀, 예측 가능한 상태 흐름

```bash
npm install @reduxjs/toolkit react-redux
```

```typescript
// store/counterSlice.ts
import { createSlice, PayloadAction } from '@reduxjs/toolkit';

interface CounterState { value: number }
const initialState: CounterState = { value: 0 };

const counterSlice = createSlice({
  name: 'counter',
  initialState,
  reducers: {
    increment: (state) => { state.value += 1; },       // Immer로 불변성 자동
    decrement: (state) => { state.value -= 1; },
    incrementByAmount: (state, action: PayloadAction<number>) => {
      state.value += action.payload;
    },
  },
});

export const { increment, decrement, incrementByAmount } = counterSlice.actions;
export default counterSlice.reducer;

// store/index.ts
import { configureStore } from '@reduxjs/toolkit';
import counterReducer from './counterSlice';

export const store = configureStore({
  reducer: { counter: counterReducer },
});

export type RootState = ReturnType<typeof store.getState>;
export type AppDispatch = typeof store.dispatch;

// 컴포넌트
import { useSelector, useDispatch } from 'react-redux';
import { increment } from './counterSlice';

function Counter() {
  const count = useSelector((state: RootState) => state.counter.value);
  const dispatch = useDispatch<AppDispatch>();
  return <button onClick={() => dispatch(increment())}>{count}</button>;
}
```

**RTK Query (API 캐싱 내장)**:

```typescript
import { createApi, fetchBaseQuery } from '@reduxjs/toolkit/query/react';

const api = createApi({
  reducerPath: 'api',
  baseQuery: fetchBaseQuery({ baseUrl: '/api' }),
  endpoints: (builder) => ({
    getUser: builder.query<User, string>({ query: (id) => `/user/${id}` }),
    updateUser: builder.mutation<User, Partial<User>>({
      query: (body) => ({ url: '/user', method: 'PUT', body }),
    }),
  }),
});

export const { useGetUserQuery, useUpdateUserMutation } = api;
// 자동 로딩/에러/캐시 상태 관리
```

### D. Context API — 라이브러리 없이 테마·언어·간단한 전역값

```typescript
// ThemeContext.tsx
import { createContext, useContext, useState, ReactNode } from 'react';

type Theme = 'light' | 'dark';
interface ThemeContextType { theme: Theme; toggle: () => void }

const ThemeContext = createContext<ThemeContextType | null>(null);

export function ThemeProvider({ children }: { children: ReactNode }) {
  const [theme, setTheme] = useState<Theme>('light');
  const toggle = () => setTheme((t) => (t === 'light' ? 'dark' : 'light'));
  return (
    <ThemeContext.Provider value={{ theme, toggle }}>
      {children}
    </ThemeContext.Provider>
  );
}

export function useTheme() {
  const ctx = useContext(ThemeContext);
  if (!ctx) throw new Error('useTheme must be used within ThemeProvider');
  return ctx;
}
```

**주의**: Context value가 바뀌면 해당 Context를 구독하는 **모든 컴포넌트가 리렌더**됨. 자주 바뀌는 상태에는 부적합.

## 라이브러리 비교표

| | **Zustand** | **Jotai** | **Redux Toolkit** | **Context API** |
|--|-------------|-----------|-------------------|-----------------|
| 학습 곡선 | 낮음 | 낮음 | 높음 | 없음 |
| 보일러플레이트 | 적음 | 매우 적음 | 중간 | 없음 |
| 렌더링 최적화 | 선택적 구독 | atom 단위 | selector | 전체 구독자 리렌더 |
| 미들웨어 | persist, devtools | jotai-tanstack, devtools | RTK Query 내장 | 없음 |
| devtools | Redux DevTools | Jotai DevTools | Redux DevTools | 없음 |
| 팀 규모 | 중소 | 중소~대 | 대규모 | 소규모 |
| 번들 크기 | ~3KB | ~3KB | ~15KB | 0 |

## 결정 가이드

```
상황 1: 사이드 프로젝트, 중간 규모, 빠른 세팅
  → Zustand (보일러플레이트 최소, devtools, persist 즉시 사용)

상황 2: 복잡한 UI, 많은 독립 상태 요소, 렌더링 최적화 중요
  → Jotai (atom 단위 구독으로 불필요한 리렌더 제거)

상황 3: 대규모 팀, 금융/커머스, 엄격한 상태 변경 추적 필요
  → Redux Toolkit (단방향 데이터 흐름 강제, RTK Query로 서버 상태 통합)

상황 4: 외부 라이브러리 없이 테마·다국어·인증 정보만 전달
  → Context API (단순, 자주 안 바뀌는 값에만)
```

## Pitfalls

- **Zustand 선택적 구독 누락**: `useStore()` 전체 상태를 구독하면 모든 변경에 리렌더 → 반드시 selector로 필요한 값만
- **Context 과용**: 자주 바뀌는 상태(폼 입력, 스크롤 위치)를 Context에 넣으면 하위 전체 리렌더 → Zustand/Jotai로
- **RTK 없이 Redux 사용**: 바닐라 Redux는 불변성 직접 관리 + 액션 타입 상수 + switch문 → RTK로 대체
- **서버 상태 혼합**: API 캐싱은 React Query나 RTK Query로 분리 — 클라이언트 상태와 서버 상태를 같은 스토어에 넣으면 캐시 무효화 복잡해짐
