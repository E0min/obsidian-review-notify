# 상태관리 심화 패턴

> TL;DR: Optimistic Update로 즉각 피드백, State Machine으로 불가능한 상태 차단, Lifting State Up vs Colocation 원칙으로 상태를 적절한 위치에 배치한다.

---

## 1. Optimistic Update (낙관적 업데이트)

서버 응답 전에 UI를 미리 변경 → 즉각적인 피드백.

### 기본 패턴 (useState)

```typescript
function LikeButton({ postId, initialLikes }: Props) {
  const [likes, setLikes] = useState(initialLikes);
  const [isLiked, setIsLiked] = useState(false);

  async function handleLike() {
    // 1. 즉시 UI 업데이트 (낙관적)
    setLikes((prev) => prev + 1);
    setIsLiked(true);

    try {
      // 2. 서버 요청
      await likePost(postId);
    } catch {
      // 3. 실패 시 롤백
      setLikes((prev) => prev - 1);
      setIsLiked(false);
      toast.error('좋아요 실패');
    }
  }

  return <button onClick={handleLike}>{likes} 좋아요</button>;
}
```

## 2. State Machine (유한 상태 기계)

복잡한 상태 전환에서 **불가능한 상태를 타입으로 차단**하는 패턴.

### 문제: boolean 플래그 폭발

```typescript
// ❌ 독립 boolean들 — 불가능한 조합이 타입상 허용
const [isLoading, setIsLoading] = useState(false);
const [isSuccess, setIsSuccess] = useState(false);
const [isError, setIsError] = useState(false);
// isLoading && isSuccess === true? 불가능한 상태인데 타입은 허용
```

### 해결: 단일 상태 열거형

```typescript
type FetchState =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: User }
  | { status: 'error'; message: string };

function UserFetcher({ userId }: { userId: string }) {
  const [state, setState] = useState<FetchState>({ status: 'idle' });

  async function load() {
    setState({ status: 'loading' });
    try {
      const user = await fetchUser(userId);
      setState({ status: 'success', data: user });
    } catch (err) {
      setState({ status: 'error', message: (err as Error).message });
    }
  }

  // switch로 모든 상태 커버
  switch (state.status) {
    case 'idle':    return <button onClick={load}>로드</button>;
    case 'loading': return <Spinner />;
    case 'success': return <div>{state.data.name}</div>; // data 타입 안전
    case 'error':   return <div>{state.message}</div>;   // message 타입 안전
  }
}
```

### useReducer로 전환 제어

```typescript
type State =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: User }
  | { status: 'error'; error: string };

type Action =
  | { type: 'FETCH' }
  | { type: 'SUCCESS'; payload: User }
  | { type: 'ERROR'; payload: string }
  | { type: 'RESET' };

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'FETCH':   return { status: 'loading' };
    case 'SUCCESS': return { status: 'success', data: action.payload };
    case 'ERROR':   return { status: 'error', error: action.payload };
    case 'RESET':   return { status: 'idle' };
    default: return state;
  }
}
```

## 3. Lifting State Up vs Colocation

### Colocation (상태를 사용하는 곳 가까이)

```typescript
// ✅ 상태를 필요한 컴포넌트에 가장 가깝게
function SearchBar() {
  const [query, setQuery] = useState(''); // SearchBar만 사용 → 여기에
  return <input value={query} onChange={(e) => setQuery(e.target.value)} />;
}
```

### Lifting State Up (공유가 필요할 때만 올리기)

```typescript
// SearchBar와 SearchResults가 같은 query를 공유할 때만 올림
function SearchPage() {
  const [query, setQuery] = useState(''); // 공통 부모로 올라옴

  return (
    <>
      <SearchBar query={query} onChange={setQuery} />
      <SearchResults query={query} />
    </>
  );
}
```

**원칙**: 항상 최대한 아래에 두고, 공유가 필요할 때만 올린다. 처음부터 전역 store로 올리지 말 것.

## 4. Compound Component — 암묵적 상태 공유

Context를 활용해 부모-자식 간 상태를 API 없이 공유:

```typescript
// Tabs 컴포넌트 — activeTab을 내부적으로 관리
const TabsContext = createContext<{ active: string; setActive: (id: string) => void } | null>(null);

function Tabs({ children, defaultTab }: { children: ReactNode; defaultTab: string }) {
  const [active, setActive] = useState(defaultTab);
  return (
    <TabsContext.Provider value={{ active, setActive }}>
      <div>{children}</div>
    </TabsContext.Provider>
  );
}

Tabs.Tab = function Tab({ id, children }: { id: string; children: ReactNode }) {
  const { active, setActive } = useContext(TabsContext)!;
  return (
    <button
      onClick={() => setActive(id)}
      style={{ fontWeight: active === id ? 'bold' : 'normal' }}
    >
      {children}
    </button>
  );
};

Tabs.Panel = function Panel({ id, children }: { id: string; children: ReactNode }) {
  const { active } = useContext(TabsContext)!;
  return active === id ? <div>{children}</div> : null;
};

// 사용
<Tabs defaultTab="overview">
  <Tabs.Tab id="overview">개요</Tabs.Tab>
  <Tabs.Tab id="details">상세</Tabs.Tab>
  <Tabs.Panel id="overview">개요 내용</Tabs.Panel>
  <Tabs.Panel id="details">상세 내용</Tabs.Panel>
</Tabs>
```

## 5. Custom Hook으로 상태 로직 추출

```typescript
// 재사용 가능한 토글 훅
function useToggle(initialValue = false) {
  const [value, setValue] = useState(initialValue);
  const toggle = useCallback(() => setValue((v) => !v), []);
  const setTrue = useCallback(() => setValue(true), []);
  const setFalse = useCallback(() => setValue(false), []);
  return { value, toggle, setTrue, setFalse };
}

// 재사용 가능한 비동기 상태 훅
function useAsync<T>(asyncFn: () => Promise<T>, deps: unknown[] = []) {
  const [state, setState] = useState<{
    data: T | null;
    loading: boolean;
    error: Error | null;
  }>({ data: null, loading: true, error: null });

  useEffect(() => {
    let cancelled = false;
    setState({ data: null, loading: true, error: null });

    asyncFn().then(
      (data) => { if (!cancelled) setState({ data, loading: false, error: null }); },
      (error) => { if (!cancelled) setState({ data: null, loading: false, error }); }
    );

    return () => { cancelled = true; };
  }, deps);

  return state;
}
```

## 6. 전역 상태 초기화 패턴

앱 부팅 시 서버/localStorage에서 상태 로드:

```typescript
// Zustand persist는 자동이지만, 커스텀 초기화가 필요한 경우
const useAuthStore = create<AuthStore>()((set) => ({
  user: null,
  isInitialized: false,
  initialize: async () => {
    const token = localStorage.getItem('token');
    if (!token) {
      set({ isInitialized: true });
      return;
    }
    try {
      const user = await validateToken(token);
      set({ user, isInitialized: true });
    } catch {
      localStorage.removeItem('token');
      set({ isInitialized: true });
    }
  },
}));

// App.tsx에서 최초 1회 실행
function App() {
  const initialize = useAuthStore((s) => s.initialize);
  const isInitialized = useAuthStore((s) => s.isInitialized);

  useEffect(() => { initialize(); }, []);

  if (!isInitialized) return <SplashScreen />;
  return <Router />;
}
```

## 7. 상태 동기화 — URL ↔ 전역 상태

```typescript
// URL 파라미터와 store 동기화
function useUrlSync() {
  const router = useRouter();
  const searchParams = useSearchParams();
  const setFilter = useFilterStore((s) => s.setFilter);

  // URL → Store (초기 로드 + URL 변경 시)
  useEffect(() => {
    const status = searchParams.get('status') ?? 'all';
    const page = Number(searchParams.get('page') ?? '1');
    setFilter({ status, page });
  }, [searchParams]);

  // Store → URL (store 변경 시)
  const filter = useFilterStore((s) => s.filter);
  useEffect(() => {
    const params = new URLSearchParams();
    params.set('status', filter.status);
    params.set('page', String(filter.page));
    router.replace(`?${params.toString()}`);
  }, [filter]);
}
```

## Pitfalls

- **Optimistic Update 롤백 누락**: 에러 catch 없이 낙관적 업데이트 → 서버/클라이언트 상태 불일치 영구화
- **State Machine 없이 boolean 조합**: `isLoading && isError === true` → 불가능한 상태가 UI에 도달
- **너무 이른 Lifting**: "나중에 쓸 것 같아서" 올리면 prop drilling 대신 over-engineering
- **Custom Hook에서 무한 루프**: `useEffect` deps 잘못 지정하면 무한 재실행
