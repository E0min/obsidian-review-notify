# 서로소 유니온 타입 (Discriminated Union)

> TL;DR: 공통 태그 필드(`type`, `kind`)로 유니온 멤버를 구별하는 패턴. optional props 지옥을 제거하고, switch 망라성 검사(`never`)로 새 케이스 추가 시 컴파일 에러로 감지한다.

---

## What

유니온 타입에서 각 멤버가 **서로 겹치지 않는(서로소)** 공통 리터럴 필드를 가지도록 설계해, 컴파일러가 해당 필드로 정확히 타입을 구별할 수 있게 하는 패턴.

"판별 유니온(Discriminated Union)" = "태그드 유니온(Tagged Union)" = "서로소 유니온" — 같은 개념.

---

## Why it matters — optional props 함정

### ❌ optional props로 표현하면 타입이 부정확해진다

```typescript
// 나쁜 설계: 세 가지 상태를 optional로 표현
interface ApiState {
  isLoading: boolean;
  data?: string;    // success일 때만 있어야 하는데 항상 선택적
  error?: string;   // error일 때만 있는데 항상 선택적
}

// data와 error가 동시에 있는 불가능한 상태도 표현 가능 → 타입이 거짓말
const state: ApiState = { isLoading: false, data: '결과', error: '오류' }; // 타입상 OK지만 의미없음
```

### ✅ 서로소 유니온으로 표현하면 불가능한 상태를 타입 레벨에서 차단

```typescript
// 좋은 설계: 각 상태를 독립된 인터페이스로
interface LoadingState { status: 'loading' }
interface SuccessState { status: 'success'; data: string }     // data 필수
interface ErrorState   { status: 'error';   message: string }  // message 필수

type ApiState = LoadingState | SuccessState | ErrorState;

// data와 error가 동시에 있는 상태 불가능 → 타입이 현실을 정확히 반영
const success: ApiState = { status: 'success', data: '결과' };    // OK
const error: ApiState   = { status: 'error', message: '오류' };   // OK
// const invalid: ApiState = { status: 'success', message: '오류' }; // 에러!
```

---

## How

### switch + narrowing

```typescript
function renderApiState(state: ApiState) {
  switch (state.status) {
    case 'loading':
      return <Spinner />;
    case 'success':
      // 여기서 state: SuccessState → state.data 접근 가능
      return <div>{state.data}</div>;
    case 'error':
      // 여기서 state: ErrorState → state.message 접근 가능
      return <div className="error">{state.message}</div>;
  }
}
```

### 망라성 검사 (Exhaustiveness Check) — `never` 활용

새로운 케이스가 추가됐는데 처리를 빠뜨리면 컴파일 에러로 감지.

```typescript
function assertNever(value: never): never {
  throw new Error(`Unhandled case: ${JSON.stringify(value)}`);
}

function renderApiState(state: ApiState) {
  switch (state.status) {
    case 'loading': return <Spinner />;
    case 'success': return <div>{state.data}</div>;
    case 'error':   return <div>{state.message}</div>;
    default:
      // 모든 케이스를 처리했으면 state는 never → 에러 없음
      // 새 케이스 추가 후 처리 안 하면 → 컴파일 에러 발생
      return assertNever(state);
  }
}

// 이후 NetworkErrorState = { status: 'network-error' }를 ApiState에 추가하면
// → default 분기에서 state가 never가 아님 → 컴파일 에러로 처리 누락 즉시 감지
```

### 실전 예시 — 도형 넓이 계산

```typescript
interface Circle    { kind: 'circle';    radius: number }
interface Rectangle { kind: 'rectangle'; width: number; height: number }
interface Triangle  { kind: 'triangle';  base: number;  height: number }

type Shape = Circle | Rectangle | Triangle;

function getArea(shape: Shape): number {
  switch (shape.kind) {
    case 'circle':
      return Math.PI * shape.radius ** 2;
    case 'rectangle':
      return shape.width * shape.height;
    case 'triangle':
      return (shape.base * shape.height) / 2;
    default:
      return assertNever(shape); // 새 도형 추가 시 여기서 에러
  }
}
```

### 중첩 판별 유니온

```typescript
interface UserAction {
  type: 'user';
  payload: { action: 'login'; userId: string } | { action: 'logout' };
}

interface SystemAction {
  type: 'system';
  payload: { code: number; message: string };
}

type AppAction = UserAction | SystemAction;

function handleAction(action: AppAction) {
  switch (action.type) {
    case 'user':
      // action.payload: { action: 'login'; userId: string } | { action: 'logout' }
      if (action.payload.action === 'login') {
        console.log(action.payload.userId); // OK
      }
      break;
    case 'system':
      console.log(action.payload.code, action.payload.message);
      break;
  }
}
```

## Pitfalls

- **태그 필드는 리터럴 타입이어야 함**: `status: string`이 아닌 `status: 'loading'` (리터럴) — 범용 string이면 narrowing이 작동하지 않음
- **공통 태그 필드명 통일**: `type`, `kind`, `status` 중 프로젝트 내에서 하나로 통일 (섞으면 혼란)
- **optional prop 함정**: 판별 유니온 대신 optional `?` 필드로 표현하면 불가능한 상태가 허용됨 → 판별 유니온으로 명확하게
- **`never` 망라성 없이 default만**: `default: throw new Error(...)` 단독으로도 런타임 보호는 되지만 컴파일 타임 감지가 안 됨 → `assertNever` 패턴 필수
