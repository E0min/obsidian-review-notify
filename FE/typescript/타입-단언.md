# 타입 단언 (Type Assertion)

> TL;DR: `as Type`으로 컴파일러에게 "이 타입으로 봐"라고 강제 지정. non-null assertion `!`은 null/undefined 아님을 강제. `as const`는 값을 리터럴 타입으로 고정. 모두 런타임 영향 없음 — 타입 에러를 숨기는 탈출구이므로 좁게 사용.

---

## What

TypeScript 컴파일러의 타입 추론을 개발자가 강제로 덮어쓰는 문법. 컴파일러보다 개발자가 더 정확히 알 때 사용. **런타임에 아무 코드도 생성되지 않는다** — 순수 타입 정보만 변경.

---

## How

### 1. `as Type` — 타입 단언

```typescript
// 기본 사용
const input = document.getElementById('search') as HTMLInputElement;
input.value; // HTMLElement에는 없지만, HTMLInputElement에는 있음 → OK

// 유니온에서 구체 타입으로
const val: string | number = getValue();
const str = val as string; // 개발자가 string임을 보장
str.toUpperCase();          // OK

// JSX에서는 <Type> 문법 대신 as 사용 (JSX와 충돌)
// const input = <HTMLInputElement>element; // JSX 파일에서 에러
const input2 = element as HTMLInputElement; // OK
```

**단언 규칙**: A를 B로 단언하려면 A와 B 중 하나가 다른 하나의 부분 집합이어야 함.

```typescript
// string → number 직접 불가 (서로 관련 없는 타입)
const n = 'hello' as number; // 에러: string → number 변환 불가

// 우회: 먼저 unknown/any로 (위험)
const n2 = 'hello' as unknown as number; // 컴파일 통과, 런타임 위험
```

### 2. `!` — Non-null Assertion

`null | undefined` 가능성을 제거. "절대 null 아니다"를 컴파일러에게 보장.

```typescript
// querySelector 반환값은 Element | null
const button = document.querySelector('.submit-btn')!; // HTMLElement (null 제거)
button.addEventListener('click', handler);              // OK

// optional chaining vs non-null assertion
const name = user?.name;   // string | undefined (안전)
const name2 = user!.name;  // string (null이면 런타임 에러 발생)

// DOM 조작처럼 확실히 존재하는 경우에만 사용
function getElement(id: string): HTMLElement {
  return document.getElementById(id)!; // id가 항상 존재함을 보장할 때
}
```

**주의**: `!`는 런타임 null 체크를 하지 않음. null이면 `TypeError` 발생.

### 3. `as const` — const assertion

값을 변경 불가 리터럴 타입으로 고정. 배열은 `readonly`, 객체는 모든 프로퍼티가 `readonly` + 리터럴 타입.

```typescript
// as const 없이 — 넓은 타입 추론
const direction = 'left';           // string
const directions = ['left', 'right']; // string[]
const config = { port: 3000 };        // { port: number }

// as const 있으면 — 리터럴 타입 고정
const direction2 = 'left' as const;           // 'left'
const directions2 = ['left', 'right'] as const; // readonly ['left', 'right']
const config2 = { port: 3000 } as const;        // { readonly port: 3000 }
```

**실전 1 — 상수 배열에서 타입 추출**:

```typescript
const ROLES = ['admin', 'user', 'guest'] as const;
type Role = typeof ROLES[number]; // 'admin' | 'user' | 'guest'

function checkRole(role: Role) { ... }
checkRole('admin'); // OK
checkRole('unknown'); // 에러
```

**실전 2 — 객체 리터럴 상수**:

```typescript
const HTTP_METHODS = {
  GET: 'GET',
  POST: 'POST',
  PUT: 'PUT',
  DELETE: 'DELETE',
} as const;

type HttpMethod = typeof HTTP_METHODS[keyof typeof HTTP_METHODS];
// 'GET' | 'POST' | 'PUT' | 'DELETE'
```

**실전 3 — 튜플 타입 고정**:

```typescript
function useState<T>(initial: T): [T, (val: T) => void] {
  // ...
}

// as const로 튜플 반환 타입 고정
function useToggle() {
  const [on, setOn] = useState(false);
  return [on, setOn] as const; // readonly [boolean, (val: boolean) => void]
}

const [isOpen, setIsOpen] = useToggle(); // 정확한 튜플 타입
```

## 세 가지 비교

| | `as Type` | `!` (non-null) | `as const` |
|--|-----------|----------------|------------|
| 역할 | 타입 강제 변환 | null/undefined 제거 | 리터럴 타입 고정 |
| 런타임 영향 | 없음 | 없음 (null이면 에러) | 없음 |
| 위험도 | 높음 | 중간 | 낮음 (readonly) |
| 주 사용처 | DOM, 라이브러리 타입 | DOM 엘리먼트, 확실한 optional | 상수 배열/객체, 튜플 |

---

## Pitfalls

- **`as any` 남용**: 타입 에러를 전부 `as any`로 숨기면 TypeScript를 쓰는 의미 없음 → 근본 원인 해결 또는 narrowing 우선
- **`!` 남용**: 추후 null이 될 수 있는 값에 `!`를 쓰면 런타임 `TypeError` → optional chaining `?.` 우선
- **`as const` vs `Object.freeze`**: `as const`는 타입 레벨만, 런타임에서 변경 가능. 런타임에서도 불변성이 필요하면 `Object.freeze()` 추가
- **이중 단언 `as unknown as Type`**: 관련 없는 타입 간 변환 우회에 쓰이지만 타입 안전성 완전 포기 — 최후의 수단으로만
