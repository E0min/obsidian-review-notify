# 에러/이슈 기록 워크플로

> TL;DR: 에러를 마주치면 즉시 기록 → 원인 분석 → 해결 → 포스팅 후보 판단. 기록하지 않으면 같은 에러를 다시 만난다.

---

## What

개발 중 마주치는 에러·이슈를 체계적으로 기록하고, 반복 방지 + 블로그 포스팅 소재로 활용하는 개인 워크플로.

**핵심 원칙**: "에러는 자산이다" — 해결한 에러는 동일한 문제를 가진 다른 개발자를 도울 수 있는 콘텐츠다.

---

## How — 에러 기록 템플릿

```markdown
## 에러명 / 제목
날짜: YYYY-MM-DD
프로젝트: 
환경: (OS, Node, 라이브러리 버전)

### 증상
> 어떤 현상이 발생했는가? 에러 메시지 전체 붙여넣기.

### 원인
> 왜 발생했는가? (근본 원인까지 파고들기)

### 해결책
> 어떻게 해결했는가? 코드/커맨드 포함.

### 재발 방지
> 다음에 같은 상황에서 어떻게 하면 빠르게 해결할 수 있는가?

### 포스팅 가치
> [ ] 블로그 후보  [ ] TIL만  [ ] 개인 메모만
```

---

## 기록 수준 분류

| 수준 | 기준 | 위치 |
|------|------|------|
| 🔴 주요 이슈 | 2시간+ 소요, 공식 문서에 없는 해법 | `Blog/drafts/` → 포스팅 |
| 🟡 일반 에러 | 30분~2시간, 재발 가능성 있음 | `_journal/YYYY-MM-DD-에러-*.md` |
| 🟢 단순 실수 | 10분 이내, 반복 않을 것 | 인라인 주석 또는 git commit message |

---

## 포스팅 전환 기준

다음 조건 중 2개 이상이면 블로그 포스팅 후보:

```
□ 검색해도 한국어 자료가 없거나 부족
□ 에러 메시지만으로는 원인 파악 불가능 (misleading)
□ 공식 문서의 설명이 불충분
□ 특정 버전 조합에서만 발생하는 이슈
□ 해결 과정이 5단계 이상
```

## 연동 도구

- **Sentry**: 프로덕션 에러 자동 캡처 → `[[../BE/observability/2026-05-20-sentry-posthog-usage]]`
- **PostHog**: 사용자 행동 이상 탐지
- **GitHub Issues**: 버그 재현 단계 + 환경 기록
- **이 노트의 일지**: 로컬 개발 환경 에러 수동 기록

---

## Pitfalls

- **에러 메시지만 복사하고 컨텍스트 생략**: 6개월 후 보면 "이게 뭐였지?" → 반드시 환경·증상·원인 모두 기록
- **해결 후 기록 안 하기**: 해결한 직후가 가장 기억이 생생할 때 — 바로 기록
- **포스팅 부담으로 기록 안 하기**: TIL 수준으로 먼저 기록, 나중에 확장
