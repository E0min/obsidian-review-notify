# AI 시대, 엔지니어는 무엇으로 살아남는가 — 의사결정의 해상도

지난 한 달간 Claude 맥스 요금제로 실무에서 코드를 짜고 검토하며 보냈다. 신기했던 건 생산성이 높아졌다는 만족보다, "그래서 앞으로 개발자는 뭐로 먹고 살아야 하나"라는 불안이 더 강하게 남았다는 점이다. AI가 코드를 대신 짜 주는 시대에 학습 방향, 성장의 본질, 그리고 개발자라는 정의 자체가 어떻게 바뀌고 있는지 한 달간 정리한 생각을 적는다.

---

## 우리는 정말 엔지니어링을 하고 있을까

과거에는 API 문서를 누구보다 빠르게 뒤지고 언어의 복잡한 문법을 외우는 능력이 곧 실력이었다. 이제 AI는 인류 역사상 그 누구보다 방대한 API 공식 문서를 완벽하게 알고 있다. 에디터에서 타이핑을 시작하기도 전에 Copilot이 수십 줄의 보일러플레이트 코드를 자동 완성해 버린다. 단순 구현 역량의 가치는 과거에 비해 0에 수렴했다. 대신 비즈니스의 전체적인 맥락을 파악하고 수많은 기술적 선택지 중 최적의 선택을 내리는 의사결정의 가치가 그 어느 때보다 중요해진 시대라는 생각이 든다.

의사결정의 맥락에서 바라보면, 코드를 짜서 무언가 그럴듯하게 동작하게 만들었다고 스스로를 엔지니어라고 부르기엔 부족하다. 돈과 시간, 서버 리소스가 무한하다면 누구나 완벽하고 견고한 마이크로서비스 아키텍처를 구축할 것이다. 진정한 엔지니어라면 제한된 자원 속에서 문제를 해결하며 데드라인을 맞춰내야 한다.

트렌드에 휩쓸려 Next.js의 App Router와 Server Component를 무작정 도입하는 건 코딩이다. 반면 "Server Component를 쓰면 자바스크립트 번들 사이즈를 대폭 줄이고 SEO 점수를 끌어올릴 수 있다. 하지만 현재 우리 팀의 Node 기반 서버 운영 경험이 턱없이 부족하고, 유저 인터랙션이 잦아 클라이언트 상태(Zustand)와 충돌하는 레거시 코드가 방대하다. 당장 전면 도입하는 것은 서비스 마비 리스크가 너무 크니, 이번 분기에는 마케팅 랜딩 페이지만 App Router로 분리하고 메인 서비스는 Pages Router를 유지하자"라고 결론 내리는 것이 진짜 엔지니어링이다.

## AI에게 질문하는 해상도를 극도로 높이기

AI에게 "React 상태 관리 라이브러리를 추천해 줘"라고 물어보자. 아마 Redux나 Zustand의 가장 기본적인 공식 문서 튜토리얼 코드를 그대로 긁어올 것이다. 우리 비즈니스가 1초에 수십 번씩 업데이트되는 실시간 웹소켓 데이터를 다루는지, 렌더링 최적화가 필수적인 무거운 폼 상태가 많은지 전혀 신경 쓰지 않는다. 획기적인 대안을 끈질기게 요구해도 결국 비슷한 코드를 변수명과 폴더 구조만 바꿔서 던져준다.

이건 AI 모델이 피드백을 통해 훈련받는 구조적 특성 때문이다. 수많은 검수자가 "이 답변이 가장 안전하고 불쾌감을 주지 않아"라고 합의한 데이터로 학습한다. 이 안전함이 엔지니어링 관점에서는 곧 평범함을 의미한다.

게다가 LLM은 내부적으로 코드의 문맥을 깊이 이해하고 짜는 것이 아니라, 프롬프트 다음에 올 확률적으로 가장 자연스러운 단어를 수학적으로 계산해 이어 붙인다. 깃허브에 널려 있는 대중적이고 뻔한 코드일수록 학습 데이터에 압도적으로 많기 때문에, 항상 가장 보편적인 정답을 우선적으로 뱉어낸다. AI가 주는 이 평균적인 코드에 만족하는 순간, 개발자 본인도 평균적인 인력으로 전락하고 만다.

가상 DOM으로 생태계를 뒤집은 React의 등장이나 Hooks 패러다임 같은, 판을 깨는 혁신을 AI가 제안할 수 있을까. 거의 불가능에 가깝다. AI는 과거 데이터의 거울이다. 기존 데이터 사이의 빈칸을 정교하게 채우는 보간(interpolation)은 기가 막히게 해내지만, 아예 존재하지 않던 새로운 차원의 축을 만들어내는 외삽(extrapolation)은 본질적으로 약하다. 문제 해결의 새로운 패러다임은 여전히 엔지니어의 몫이라는 사실을 잊지 말아야 한다.

AI에게 아무리 "코드를 더 깔끔하게 리팩토링해 줘"라고 명령해도 결과가 고만고만한 이유는 질문의 해상도가 낮기 때문이다. 에러가 났을 때 눈에 보이는 현상만 던져서는 안 된다. 철저하게 구체적인 상황을 부여해야 한다. AI의 결과물 퀄리티는 프롬프트를 작성하는 엔지니어의 도메인 지식 수준과 정확히 비례한다.

> **코더의 질문**: "Next.js에서 API 데이터 가져오는 리스트 코드 짜 줘. 스크롤할 때 자꾸 버벅거려."
>
> **엔지니어의 질문**: "Next.js 14 App Router 환경에서 사용자 대시보드를 만들고 있어. 데이터는 외부 CMS에서 가져오는데 실시간성이 중요하지 않아서 1시간 단위로 캐싱(ISR)하고 싶어. fetch API의 `next: { revalidate: 3600 }` 옵션을 사용해서 컴포넌트 레벨에서 데이터를 패칭하는 코드를 짜 줘. 스크롤 성능을 위해 React의 `useDeferredValue`를 적용하고, 에러가 났을 때 보여줄 `error.tsx` 바운더리 처리도 반드시 포함해."

차이가 명확히 느껴지는가. 두 번째 질문은 AI에게 코딩을 부탁하는 것이 아니라, 이미 머릿속에 완성된 뼈대(아키텍처)의 빈칸을 채우라고 지시하는 것에 가깝다.

## AI에게 무엇을 질문할 것인가

가장 흔히 저지르는 실수는 AI에게 "로그인 페이지 화면 코드를 짜 줘"라고 묻는 일이다. AI를 단순한 코드 자판기가 아닌, 복잡한 아키텍처를 논의하고 비용을 산정하는 파트너로 대우하려면 다음 세 가지 원칙을 기억해야 한다.

첫째, 코딩이 아닌 의사결정의 도움 도구로 사용하기. AI에게 코드를 짜 달라고 요구하지 말고 의사결정을 도와 달라고 물어본다. AI는 훌륭한 구현 전문가지만 비즈니스의 맥락은 모른다. 단순히 기능을 만들어 달라고 요구하면 전 세계의 튜토리얼에서 긁어모은 평균적인 코드를 조합해서 던져줄 뿐이다. 레거시 시스템과의 의존성, 렌더링 60fps 보장 같은 가장 까다로운 제약 조건을 입력해야 한다. 맥락(Context), 제약 조건(Constraint), 미래의 변수를 명확히 제시하며 질문해야 의사결정에 필요한 기술적 트레이드오프를 논의할 수 있다.

둘째, How가 아닌 Why와 Which를 집요하게 파고들기. AI에게 물어야 할 것은 "어떻게 구현할 것인가"가 아니라 "왜 이 기술을 써야 하는가"와 "어떤 대안이 최선인가"다. 구현 로직을 묻기 전에 시스템 설계와 트레이드오프를 먼저 논의해야 한다.

셋째, 비교 검토를 위한 구체적인 시나리오를 제시하기. 단순한 추천을 넘어 정확한 장단점과 리스크를 비교하도록 구체적인 제약 조건을 던져야 한다.

> "React Context API로 전역 상태를 관리하는 것과 Zustand 같은 외부 라이브러리를 도입하는 것 중, 현재 진행 중인 프로젝트 규모에서 어떤 것이 렌더링 최적화와 유지보수에 유리할까. 상태 변경 시 발생하는 불필요한 재렌더링 이슈를 중심으로 표로 비교해 줘."

> "사용자가 입력하는 검색어 자동완성 기능을 만들려고 해. 매 키 입력마다 API를 호출하면 서버 부하가 심할 텐데, 디바운싱과 쓰로틀링 중 우리 서비스의 검색 UX에 더 적합한 방식은 무엇일까. 사용자의 대기 시간과 API 호출 횟수를 기준으로 비교해 줘."

## AI의 답변을 어떻게 검증할 것인가

질문의 해상도만큼이나 답변을 다루는 해상도도 중요하다. 검증 없이 받아 적으면 결국 AI에게 코드 책임을 떠넘기는 모양이 된다.

먼저 이 코드가 어떤 기술 부채를 숨기고 있는지 의심해야 한다. AI는 당장 눈앞의 기능이 돌아가는 가장 빠른 지름길을 택한다. AI가 짠 useEffect 로직을 그대로 가져왔더니 타이머나 스크롤 이벤트 리스너를 등록만 해 두고 언마운트 시점에 cleanup 하지 않아 페이지를 이동할 때마다 콘솔에 경고가 뜨거나 브라우저 메모리가 차오르는 경우가 있다. 단순한 게시글 목록을 필터링하는데 불필요한 이중 반복문을 만들어 성능을 떨어뜨리지는 않았는지, 데이터가 10개일 때는 잘 돌아가지만 1,000개가 되었을 때 화면이 버벅거리지 않을지 역으로 의심해야 한다.

다음으로 기존 시스템의 아키텍처를 파괴하지 않는지 확인해야 한다. 우리 프로젝트의 거대한 룰을 알 리 없는 AI가 화려한 최신 유틸리티 함수를 추천한다. "지금 우리 프로젝트는 Feature-Sliced Design 아키텍처를 따르고 있는데 네가 짠 이 코드는 도메인 로직과 UI 계층을 강하게 결합시키고 있어. 이대로 가면 나중에 순환 참조 지옥에 빠지게 돼. 관심사의 분리 원칙에 맞게 당장 로직을 커스텀 훅으로 분리해라"라고 지적하며 궤도를 수정해야 한다.

마지막으로 이 코드를 동료에게 100% 설명할 수 있는지 자문해야 한다. 운영 서버가 다운되어 핫픽스를 배포해야 하는 아찔한 순간, "AI가 짜 준 거라 저도 왜 이런지 잘 모릅니다"라는 변명은 현업에서 절대 통하지 않는다. 내가 완벽하게 설명할 수 없는 코드는 언제 터질지 모르는 시한폭탄과 같다. 제너레이터 함수나 클로저처럼 작동 원리를 모르는 코드는 단 한 줄도 프로덕션 환경에 올리지 말아야 한다.

## 좋은 질문을 위해 무엇을 공부할 것인가

질문의 해상도를 높이려면 결국 두 축을 함께 키워야 한다. 비즈니스 도메인과 기술의 원리.

비즈니스와 도메인 지식은 내재화돼 있어야 한다. AI는 세상에 공개된 모든 기술 문서를 학습했지만 우리가 당면한 비즈니스의 특수성과 도메인 로직은 전혀 알지 못한다. 기술적인 완벽함이 비즈니스의 정답을 보장하지는 않는다. 예를 들어 이커머스의 장바구니 기능을 구현할 때 AI는 단순히 상품을 배열에 추가하고 가격 합계를 구하는 깔끔한 코드를 작성해 줄 것이다. 하지만 실제 도메인에서는 상품의 실시간 재고 차감 시점, 서로 다른 할인 쿠폰의 중복 적용 우선순위, 동시 접속자의 결제 충돌 처리 같은 수많은 비즈니스 엣지 케이스가 존재한다. 이러한 도메인 지식이 없으면 AI에게 어떤 제약 조건을 요구해야 할지조차 알 수 없다. 비즈니스 로직을 완벽히 꿰뚫고 있어야만 AI가 내놓은 코드가 실제 서비스 환경에서 구멍 없이 작동할 수 있는지 검증할 수 있다. 구현 로직은 AI에게 맡기더라도 해결해야 할 비즈니스의 본질은 온전히 개발자의 머릿속에 있어야 한다.

또 한 축은 원리(Under the Hood)를 파고드는 일이다. AI나 라이브러리가 마법처럼 알아서 해 줄 것이라 믿고 넘어가서는 안 된다. 프레임워크 이면에 있는 브라우저나 언어의 기본 동작 원리를 이해해야 한다. 자바스크립트의 이벤트 루프가 비동기 작업을 어떻게 처리하는지 모르면 AI가 짜 준 코드의 실행 순서가 꼬였을 때 원인을 찾을 수 없다. AI가 아무리 화려한 CSS 코드를 짜 주더라도 브라우저의 렌더링 과정(Reflow와 Repaint)을 모르면 화면이 버벅거릴 때 어디를 고쳐야 할지 막막해진다. 당장 프레임워크의 새로운 문법을 외우는 것보다 그 아래에서 동작하는 코어 언어의 특성과 웹의 기본기를 다지는 것이 훨씬 중요하다.

## AI와 치열하게 논쟁하며 지배하기

미래의 개발 생태계는 명확하게 둘로 나뉠 것이다. AI가 내놓은 결과물을 통제하고 AI를 의사결정 도구로만 활용하면서 구현은 위임하는 소수의 아키텍트와, AI가 주는 대로 코드를 복사하다 결국 그 AI에게 일자리를 고스란히 내어주는 평범한 코더로.

빠르고 그럴듯한 정답을 타이핑하는 일은 AI에게 던져 주자. 비즈니스 제약 조건 속에서 특정 기술을 왜 선택해야만 했는지 최적의 이유를 만들어 내고, 수만 명의 유저가 몰려도 끄떡없는 시스템의 뼈대를 굳건하게 책임지는 것은 오직 인간 엔지니어의 역할이다.

AI에게 질문을 던질 때 비즈니스 제약과 기술적 한계를 고려한 아키텍처 방향성을 먼저 분명하게 제시해야 한다. 그리고 AI와의 논쟁 속에서 시스템을 해부하는 훈련을 반복해야 한다. AI에게는 구현을 맡기고, 비즈니스 컨텍스트와 도메인 분야에 대한 날카로운 분석을 탑재하여, AI에게 조언을 받는 게 아니라 AI에게 시키는 사람이 되는 것. 이게 지난 한 달 동안 내가 도달한, 앞으로 살아남을 엔지니어의 자리에 대한 잠정적 답이다. 다음 한 달에 어떻게 갱신될지는 또 다른 글에서 적어 보려 한다.
