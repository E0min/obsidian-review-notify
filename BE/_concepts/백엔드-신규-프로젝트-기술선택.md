# 백엔드 신규 프로젝트 기술 선택 가이드

> TL;DR: Spring Boot 기반 신규 백엔드 프로젝트 시작 시 결정해야 할 항목 — CI/CD(GitHub Actions ✅), 컨테이너/배포(Docker + AWS EC2), 인증(OAuth2.0), 실시간(WebSocket + STOMP), 컨벤션. 각 결정의 근거와 대안을 정리.

> **원본 노트**: Notion "백엔드 (1)"은 70KB의 PR 템플릿 + 기술 리서치 + 비교 표 27개 접기 블록으로 구성된 프로젝트 종속 문서. 옵시디언에는 **재사용 가능한 결정 패턴**만 추출. 상세는 [Notion 원본](https://www.notion.so/1c835c50238380f49d04de583b4ebf95).

## 1. CI/CD 도구 선택

### 핵심 개념

- **CI (Continuous Integration)**: 코드 변경마다 자동 빌드·테스트 → 품질 유지
- **CD (Continuous Deployment)**: 검증된 코드 자동 배포 → 개발-배포 사이클 단축

### GitHub Actions vs Jenkins

| 항목 | GitHub Actions | Jenkins |
|---|---|---|
| **설치/운영** | 없음 (GitHub 통합) | 별도 서버 필요 |
| **설정** | `.github/workflows/*.yml` YAML | Jenkinsfile + UI |
| **비용** | 퍼블릭 무료 / 프라이빗 일부 무료 | 자체 호스팅 비용 |
| **플러그인** | GitHub Marketplace Actions | 1,500+ 플러그인 |
| **GitHub 연동** | 네이티브 ✅ | 별도 설정 |
| **확장성** | GitHub-hosted runner | 자체 인프라 자유 |
| **러닝 커브** | 낮음 | 높음 |

**결정**: 소규모/중규모 Spring Boot + React 프로젝트는 **GitHub Actions ✅**
- GitHub 저장소면 별도 인프라 불필요
- YAML 한 파일로 빠른 셋업
- 비용 부담 적음

**Jenkins 선택 시점**: 대규모·복잡한 파이프라인, 사내 GitLab/Bitbucket, 1,500+ 플러그인 활용 필요

### 추천 스택 (소규모)

```
GitHub Actions (CI: 빌드+테스트)
  ↓
Docker (이미지 빌드)
  ↓
AWS EC2 (배포 대상)
```

`.github/workflows/ci.yml` 예시:
```yaml
name: CI/CD
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up JDK
        uses: actions/setup-java@v4
        with: { java-version: '17', distribution: 'temurin' }
      - run: ./gradlew build
      - name: Docker build & push
        run: |
          docker build -t app:${{ github.sha }} .
          docker push ...
      - name: Deploy to EC2 (SSH)
        run: ...
```

## 2. 인증 방식 선택

| 방식 | 비용 | 추천 |
|---|---|---|
| **PASS / 문자 인증** | API 호출당 과금 | ❌ 소규모 프로젝트 비용 부담 |
| **OAuth 2.0 (카카오·구글·네이버)** | 무료 | ✅ **추천** |
| **이메일 + 비밀번호** | 무료 | ✅ 보조 |
| **JWT 직접 발급** | 무료 | ✅ Stateless API에 적합 |

**결정 패턴**:
- B2C 서비스 → OAuth 2.0 (가입 마찰 ↓)
- 이메일 + OAuth 혼합 가능
- 토큰 발급은 JWT 또는 세션 — 상세 [[../auth/_MOC]]

→ 인증 흐름: [[../../CS/network/쿠키-세션-jwt]]

## 3. 실시간 기능 — WebSocket + STOMP

채팅·알림·오디오 스트리밍이 필요할 때:

| 옵션 | 용도 |
|---|---|
| **WebSocket (raw)** | 자유로운 메시지 포맷, 직접 구현 |
| **STOMP over WebSocket** | 발행/구독 패턴, Spring 통합 ✅ |
| **Socket.io** | Node.js 진영 표준, fallback 자동 |
| **SSE (Server-Sent Events)** | 서버 → 클라이언트 단방향만 필요할 때 |
| **WebRTC** | P2P 미디어/데이터 ([[../../CS/network/webrtc]]) |

**Spring Boot + 채팅 결정**: WebSocket + STOMP
- `spring-boot-starter-websocket`로 즉시 가능
- 토픽 발행/구독 모델로 채팅방 구현 깔끔
- 시그널링이 필요한 영상 통화는 WebSocket 시그널링 + WebRTC P2P 조합

→ [[../../CS/network/websocket]] (WebSocket 기본)

## 4. 신규 프로젝트 셋업 체크리스트

```
✅ 코드 컨벤션 (Google Java Style / Airbnb JS)
✅ Git 브랜치 전략 (Git Flow / GitHub Flow / Trunk-Based)
✅ PR 템플릿 + Issue 라벨
✅ API 명세 (Swagger / OpenAPI 3.0)
☐ ERD (dbdiagram.io / cloud.erdcloud)
☐ 아키텍처 다이어그램
☐ 역할 분담 (RACI)
✅ CI/CD 파이프라인 (.github/workflows/)
✅ Docker 컨테이너화
✅ 모니터링 (Sentry, CloudWatch)
☐ 보안 (HTTPS, OWASP Top 10)
```

체크 ✅는 보통 1주 안에 셋업, ☐는 도메인 확정 후.

## 5. 기술 결정 일반 원칙

1. **익숙한 것 우선**: 팀이 잘 아는 스택이 가장 빠른 배포 → 비즈니스 검증 우선
2. **운영 비용 < 개발 시간 비용**: 작은 서비스에 K8s·MSA 같은 over-engineering 피하기
3. **무료 티어 활용**: GitHub Actions(2,000분/월), Vercel 호빙, AWS Free Tier 1년
4. **마이그레이션 가능성 열어두기**: ORM, 인증, 결제 등은 추상화 레이어 두기
5. **모니터링부터 셋업**: 배포 다음 날 장애 나면 보이는 게 없음 — Day 1부터 Sentry/로그 수집

## Pitfalls

- **JenkinsLab만 보고 도입** → 작은 팀엔 운영 부담 ↑. GitHub Actions로 충분한 경우가 많음
- **K8s 처음부터 도입** → EC2 + Docker Compose면 첫 6개월 충분. 트래픽·팀 규모 따라 결정
- **PASS/문자 인증 무계획 도입** → MAU 1만 미만은 비용 회수 안 됨
- **JWT만 쓰고 무효화 안 함** → 로그아웃·토큰 탈취 대응 어려움. Refresh Token + Redis 블랙리스트 조합
- **WebSocket을 모든 실시간에 사용** → 단방향이면 SSE가 더 가볍고 HTTP/2와 자연스럽게 동작
